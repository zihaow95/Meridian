# 正式工程规范

版本：V1.0

日期：2026-07-02

状态：已确认基线

基线确认日期：2026-07-06

项目代号：Project Meridian（经纬）

## 1. 适用范围

本文适用于`D:\Projects\Meridian`正式工程根目录下的后端、前端、数据库、测试、容器、CI和发布工作。

旧原型目录不适用本文，也不得被正式构建引用。

## 2. 基本原则

- 简单优先：首期保持模块化单体；
- 领域优先：代码按业务模块组织，不按Controller/Service/Model全局分层；
- 后端权威：权限、状态和数据校验不能只存在于前端；
- 事务明确：跨表写入必须有应用服务和事务边界；
- 历史不可变：文件、产品版本、模板、决策和快照不覆盖；
- 测试先行：业务规则先写失败测试，再写最小实现；
- 配置有版本：模板和字典发布后不可修改；
- 失败可见：禁止静默跳过、静默降级和无审计成功。

## 5. 后端模块结构

每个Django业务应用采用：

```text
apps/<domain>/
├─ models/
├─ services/
├─ queries/
├─ api/
├─ policies/
├─ events/
├─ tasks/
├─ migrations/
└─ tests/
```

规则：

- `models`表达实体、约束和简单领域行为；
- `services`编排命令、事务、权限、审计和事件；
- `queries`负责权限过滤后的读取；
- `api`只解析、调用和序列化；
- `policies`实现本领域ABAC；
- `tasks`只接收标识并重新读取权威数据；
- 模块不得直接修改其他模块模型；
- 跨域写入通过公开应用服务；
- 查询可使用明确的跨表查询服务。

单个文件超过约400行时应检查是否混合职责，但不得为追求行数机械拆分。

## 6. 应用服务规范

所有业务命令必须：

1. 解析调用主体和输入；
2. 读取当前对象；
3. 执行权限判断；
4. 建立数据库事务；
5. 在事务中锁定或验证`version_no`；
6. 再次校验业务前置条件；
7. 执行领域变化；
8. 写入审计和事件发件箱；
9. 提交后再触发外部通知。

View、Serializer、Celery任务和Vue组件不得直接实现关键状态迁移。

应用服务命名使用命令语义，例如：

- `SubmitProposal`；
- `ApproveAndCreateProject`；
- `PublishProductChangeSet`；
- `ConvertIssueToIterationProposal`。

## 7. 数据库规范

### 7.1 通用

- InnoDB；
- `utf8mb4`；
- 严格SQL模式；
- `READ COMMITTED`；
- 时间保存UTC；
- 金额、比例和计量值使用DECIMAL；
- 对外使用UUID `public_id`；
- 内部主键使用BIGINT；
- 核心业务表包含`organization_id`；
- 状态使用varchar代码，不使用MySQL ENUM。

### 7.2 外键和删除

- 正式业务对象默认`PROTECT`；
- 仅纯从属且未被历史引用的数据允许`CASCADE`；
- 已进入正式流程的业务数据不物理删除；
- 不创建全局通用软删除字段；
- 撤回、失效、Pass、退市和归档使用领域状态。

### 7.3 索引

- 每个唯一业务规则必须有数据库唯一约束；
- 列表索引以组织、状态、负责人、时间为基础；
- 外键字段建立索引；
- 组合索引顺序依据实际过滤条件；
- 新索引必须有查询或约束用途；
- 不为低基数字段单独滥建索引。

### 7.4 迁移

- 每个提交只包含本功能所需迁移；
- 数据迁移和结构迁移尽量分开；
- 禁止修改已经进入共享环境的历史迁移；
- 大表变更先评估锁表和回滚；
- 发布前运行`makemigrations --check`；
- 生产回滚不能依赖删除已写入的业务事实。

## 8. API规范

- 前缀：`/api/v1`；
- JSON、UTF-8、ISO 8601；
- 路径`{id}`统一为UUID `public_id`；
- GET查询、POST创建/命令、PATCH普通字段更新；
- 关键状态使用动作端点，不允许直接PATCH状态字段；
- 列表必须分页和稳定排序；
- 金额和高精度小数以字符串传输；
- 使用统一错误结构：`code`、`message`、`details`、`trace_id`；
- 错误码一经前端使用不得改变含义；
- OpenAPI是前端类型和契约测试来源；
- API变更必须同步测试和OpenAPI快照。

API不得返回：

- 数据库主键；
- SQL、堆栈和服务器路径；
- 密钥、令牌和完整外部错误；
- 无权字段或对象存在性。

## 9. 权限与安全

### 9.1 身份和会话

- 生产只允许钉钉企业认证；
- 开发登录只存在于开发/测试设置；
- 使用服务端会话；
- Cookie必须`HttpOnly`、`Secure`和适当`SameSite`；
- 所有写请求启用CSRF；
- 回调`state`一次性且短时；
- 停用和离职用户立即禁止新请求。

### 9.2 权限

- 默认拒绝；
- 列表先过滤，详情再判权；
- 字段、预览、下载、导出和通知分别判权；
- 平台管理权与业务数据访问权分离；
- 专项授权不得超过授权人自身范围；
- 排障访问必须限时、限对象并全量审计；
- 写命令在事务中重新判权。

### 9.3 密钥

- 密钥不得进入Git、日志、审计和发布包明文；
- 测试和生产使用不同凭据；
- `.env.example`只写变量名和说明；
- 生产密钥由服务器受限文件或环境注入；
- 凭据轮换不修改业务代码。

## 10. 文件规范

- 文件二进制保存在NAS，不进入MySQL；
- 对象键使用UUID，不使用原文件名；
- 上传流式写入，不整体加载内存；
- 完成上传时计算SHA-256；
- 使用PENDING—ACTIVE补偿流程；
- 只有ACTIVE文件可被业务引用；
- 锁定、受控和决策引用版本不可覆盖；
- NAS不直接暴露给用户；
- 下载通过短时、单用途票据和Nginx内部路径；
- 预览与原件下载分别授权；
- 临时和导出文件按保留策略清理。

## 11. 异步任务

- Redis不是业务事实来源；
- 可靠事件先写MySQL发件箱；
- Celery任务参数只传ID和必要小值；
- 消费者按`event_id`幂等；
- 设置超时、有限重试和退避；
- 失败形成可查询记录；
- 重大阶段门和产品批准不能由异步任务自动决定；
- 定时任务只能执行已批准的生效计划。

## 12. 前端规范

前端按业务域组织：

```text
src/modules/<domain>/
├─ api/
├─ components/
├─ pages/
├─ stores/
├─ types/
└─ tests/
```

规则：

- API类型由OpenAPI生成；
- Pinia只保存跨页面共享状态；
- 业务状态代码使用统一字典；
- 路由守卫不替代后端权限；
- 关键提交防重复点击并处理幂等响应；
- 409冲突必须提示刷新和比较；
- 敏感信息不进入URL、localStorage和前端日志；
- 表格分页、筛选和排序由后端执行；
- 所有待办动作跳转系统页面完成。

## 13. 日志与错误处理

技术日志统一包含：

- UTC时间；
- 环境和服务；
- 日志级别；
- `trace_id`；
- 稳定事件代码；
- 必要用户和对象标识。

禁止记录密码、Cookie、令牌、文件正文、完整配方、完整请求头和无关个人数据。

业务异常返回稳定错误码；未知异常返回通用错误并保留`trace_id`。捕获异常后必须处理、转换或重新抛出，禁止空`except`。

## 14. Git规范

- 默认分支保持可部署；
- 功能分支使用`codex/`或团队确认的统一前缀；
- 一个提交只表达一个意图；
- 迁移、测试和实现应在同一功能提交中完整出现；
- 禁止提交临时日志、截图、上传文件和本地数据库；
- 合并前必须通过全量质量门禁；
- 设计基线、生产发布使用明确标签；
- 不重写已经共享的生产发布历史。

提交类型：

- `feat`：业务能力；
- `fix`：缺陷；
- `test`：测试；
- `docs`：文档；
- `refactor`：不改变行为的重构；
- `chore`：工程配置；
- `ci`：流水线；
- `release`：发布基线。

## 15. CI与发布

CI顺序：

1. 锁文件安装；
2. 后端格式、静态检查和类型检查；
3. MySQL集成测试；
4. 前端lint、类型和单元测试；
5. OpenAPI一致性检查；
6. 前端生产构建；
7. Docker镜像构建；
8. E2E冒烟；
9. 生成离线发布包和SHA-256。

CI不连接生产服务器。管理员人工部署同一个已在测试环境验证的发布包。

## 16. 资源与性能

- Docker Compose设置CPU和内存硬限制；
- 总上限6 vCPU、8GB；
- Gunicorn和Celery不按服务器总核心自动扩张；
- 列表必须分页；
- 批量导入分批处理；
- 文件流式传输；
- 看板不得在请求中执行无界全表计算；
- 日志轮转；
- 扩容依据监控数据，不依据主观估计。

## 17. 完成定义

一个功能只有同时满足以下条件才算完成：

- 对应PRD/TRD编号已标注；
- 业务规则测试先失败后通过；
- MySQL迁移可从空库执行；
- 权限、审计、并发和失败路径已测试；
- OpenAPI和前端类型同步；
- 前端关键交互可运行；
- 静态检查和全量相关测试通过；
- 没有跳过项、TODO或未披露限制；
- 文档和配置样例已更新；
- 提交范围只包含该功能相关变更。
