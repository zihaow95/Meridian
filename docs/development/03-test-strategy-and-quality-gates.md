# 测试策略与质量门禁

版本：V1.0

日期：2026-07-02

状态：已确认基线

基线确认日期：2026-07-06

项目代号：Project Meridian（经纬）

## 1. 目标

测试必须证明系统满足PRD、TRD、权限、可追溯性和运行保障要求，而不只是证明页面能打开或接口返回200。

重点风险：

- 状态机被绕过；
- 未授权访问敏感数据；
- 重复请求产生重复业务事实；
- 文件、产品版本和决策历史被覆盖；
- 跨模块事务产生半成品；
- 异步任务丢失或重复处理；
- 模板更新污染既有项目；
- 迁移和人工数据覆盖破坏来源事实；
- 发布、备份和恢复不可执行。

## 2. 测试原则

- 业务规则先写失败测试，再写最小实现；
- 测试名称表达规则存在的原因；
- 使用MySQL验证正式数据库行为；
- 权限测试覆盖允许和拒绝两面；
- 状态测试覆盖合法与非法迁移；
- 并发测试必须验证最终数据库事实；
- Mock只隔离真正的外部系统；
- 测试不能依赖执行顺序；
- 跳过、预期失败和重试必须可见；
- 没有证据不得宣称功能完成。

## 3. 需求追踪

每个测试应通过标记或测试说明关联：

- PRD编号：GLB、OPP、PIM、EXE、OPS、PLT、NFR；
- TRD章节或命令；
- 缺陷编号（回归测试时）。

CI生成需求覆盖清单：

```text
需求编号 → 测试用例 → 最近结果 → 所属发布版本
```

92项需求在首期生产发布前必须至少有一种有效证据：

- 自动化测试；
- 配置校验；
- 恢复演练；
- 明确的人工业务验收记录。

纯文档说明不构成功能验收证据。

## 4. 测试层级

### 4.1 领域单元测试

范围：

- 状态机；
- 领域不变量；
- 字段和内容校验；
- 阶段门结果映射；
- 权限策略函数；
- 指标和风险规则；
- 差异和内容哈希。

特点：

- 不访问网络；
- 尽量不访问数据库；
- 快速、确定；
- 每个业务分支有明确预期。

### 4.2 应用服务测试

使用MySQL事务验证：

- 命令编排；
- 权限重新判断；
- 审计和发件箱；
- 幂等键；
- 失败回滚；
- 跨聚合原子操作；
- 乐观锁和行锁。

关键应用服务不得只用Mock仓储测试。

### 4.3 API契约测试

验证：

- 请求和响应Schema；
- UUID `public_id`；
- 分页和稳定排序；
- 统一错误结构；
- 权限过滤；
- CSRF和会话；
- OpenAPI与真实接口一致；
- 关键状态不能直接PATCH。

### 4.4 前端单元和组件测试

使用Vitest验证：

- 状态展示；
- 表单校验；
- 权限动作隐藏；
- 错误码处理；
- 409冲突提示；
- 待办状态；
- 差异比较；
- 敏感字段不进入持久缓存。

前端权限测试只验证体验，不能替代后端权限测试。

### 4.5 E2E测试

使用Playwright通过浏览器和真实API验证关键业务闭环。E2E使用隔离MySQL数据库、Redis和临时文件目录。

### 4.6 发布与恢复测试

验证：

- Docker镜像；
- 数据库迁移；
- 离线发布包；
- 测试环境部署；
- 健康检查；
- 回滚；
- 数据库和文件联合恢复。

## 5. 后端测试目录

```text
backend/tests/
├─ unit/
├─ integration/
├─ api/
├─ permissions/
├─ concurrency/
├─ files/
├─ migrations/
└─ fixtures/
```

业务模块内可以保留紧邻实现的测试，但全局并发、迁移和跨域测试集中在上述目录。

## 6. 测试数据

- 使用工厂函数创建用户、角色、机会、项目和产品；
- 测试数据使用虚构名称，不复制真实配方、成本和个人信息；
- 每个测试只创建必要对象；
- 固定时间测试使用可控时钟；
- 文件测试使用临时目录；
- 外部系统使用契约固定响应；
- E2E场景使用稳定业务编号前缀；
- 测试结束后清理数据库、Redis和临时文件。

禁止使用生产数据直接运行自动化测试。

## 7. 核心领域测试矩阵

### 7.1 提案—立项

必须覆盖：

- 无资格用户不能正式提交；
- 普通员工只能参与联合提案；
- 四项核心内容和公开摘要校验；
- 一个提案只计一次额度；
- 评审版本锁定；
- 经管会结论与老板决策差异；
- 待补充、暂缓、Pass和复议；
- 合并和拆分来源不丢失；
- 重复立项只创建一个项目；
- 初始化失败全部回滚。

### 7.2 产品档案

- 产品—版本—SKU—渠道关系；
- 固定字段和属性Schema；
- 属性变化使确认失效；
- 营养值和标签不一致阻止发布；
- 基线变化阻止旧草稿发布；
- 发布失败保持有效档案不变；
- 并行有效范围冲突；
- 条码和外部绑定唯一；
- 存量产品重复提示和幂等导入；
- 历史版本和素材不可覆盖。

### 7.3 项目执行

- 模板快照不受新版本影响；
- 当前项目组长始终是唯一A；
- 每个任务只有一个R；
- 依赖环被拒绝；
- 核心交付物只能豁免；
- 新文件版本必须重新专业确认；
- 阶段门引用提交时版本；
- 带例外通过只限产品总监；
- 首次上市重大阶段门；
- 发布失败进入待修复；
- 逾期只提醒和标红；
- 在途项目不补造历史阶段门。

### 7.4 运营与退市

- SKU × 渠道 × 实际时间粒度；
- 人工有效值不覆盖原始事实；
- 不兼容口径不强行汇总；
- 数据不足不生成正常结论；
- 风险信号不自动创建项目；
- 经营监督人可关闭轻微信号；
- 一个信号只有一个活动主议题；
- 转迭代只创建提案草稿；
- 普通迭代结果回写；
- 退市决策和执行分离；
- 退市后历史仍可查询。

### 7.5 平台能力

- 钉钉身份不自动授予关键角色；
- 平台管理员不能读取高敏业务值；
- 项目身份只在对应项目生效；
- 专项授权不能越权且到期失效；
- 双人复核开关两种模式；
- 文件PENDING补偿；
- 短时下载票据；
- 通知失败不回滚业务；
- 审计失败回滚关键操作；
- Redis故障后发件箱事件可恢复；
- 配置V2不改变使用V1快照的项目。

## 8. 权限测试

每个敏感资源至少建立以下矩阵：

| 场景 | 预期 |
|---|---|
| 无角色、无对象身份 | 拒绝 |
| 有角色、无对象范围 | 拒绝 |
| 有范围、数据等级不足 | 拒绝 |
| 有查看权、无下载权 | 可查看，不可下载 |
| 项目成员访问其他项目 | 拒绝 |
| 平台管理员访问业务敏感值 | 拒绝 |
| 有效专项授权 | 允许授权动作 |
| 专项授权到期 | 立即拒绝 |

列表、详情、字段、预览、下载、导出和通知摘要分别测试。

拒绝结果不得泄露对象标题、文件名和存在性。

## 9. 并发与幂等测试

使用独立数据库连接和同步屏障验证：

- 两个并发提案提交；
- 两个并发重大阶段门决策；
- 两个并发项目创建；
- 两个项目同时发布同一产品基线；
- 属性编辑与专业确认并发；
- 文件上传会话重复完成；
- 同一信号并发创建活动主议题；
- 同一议题并发转提案；
- 同一人工有效值并发生效；
- 同一事件重复消费。

测试必须查询最终数据库行数、状态、唯一约束和审计记录，不能只比较HTTP响应。

## 10. 文件测试

- 空文件、超限和类型不匹配；
- 上传中断；
- 数据库成功但文件移动失败；
- 文件移动成功但ACTIVE更新失败；
- PENDING超时补偿；
- 文件缺失巡检；
- 同名和同哈希文件；
- 新版本和历史引用；
- 专业确认绑定具体版本；
- 无权预览和下载；
- 下载中授权到期；
- 导出临时文件清理；
- NAS短时不可用后的恢复。

正式历史文件不得被清理测试误删。

## 11. 集成测试

### 11.1 钉钉

- 正常认证；
- 非法或过期`state`；
- 无内部用户绑定；
- 停用/离职用户；
- 组织同步部分失败；
- 通知成功、失败、重试和去重；
- 深链接进入后重新判权。

### 11.2 外部数据

- 接口超时和认证失败；
- 重复批次；
- 部分成功；
- 未映射对象；
- 迟到数据；
- 单位和币种冲突；
- 外部系统恢复后重试；
- 错误信息不泄露凭据。

## 12. 迁移测试

- 空模板；
- 必填列缺失；
- 错误行和有效行混合；
- 重复产品、条码和外部编码；
- 部分完整产品；
- 重复运行同一批次；
- 在途项目从真实阶段继续；
- 历史文件来源标记；
- 迁移失败无半成品正式资产；
- 导入结果与源文件行数核对。

至少执行一次全量模拟迁移和一次增量重跑。

## 13. 性能与容量

在不超过6 vCPU、8GB容器上限下，以接近首期规模的数据执行：

- 200名用户；
- 10个并行项目；
- 产品、版本、SKU、渠道和文件模拟数据；
- 经营事实达到首年预估规模。

初始验收目标：

| 场景 | 目标 |
|---|---|
| 普通列表和详情API | p95 ≤ 1秒 |
| 普通业务写操作 | p95 ≤ 2秒 |
| 生命周期和产品看板 | p95 ≤ 2秒 |
| 异步待办/通知入队 | 业务提交后60秒内可见 |
| 文件上传下载 | 在配置上限内稳定流式传输 |
| 服务端错误率 | 稳态测试低于0.5% |

外部系统耗时、文件实际传输时间和长时间导入不纳入普通API目标，但必须异步化并显示进度。

## 14. 备份与恢复

上线前必须完成：

1. MySQL完整备份；
2. NAS文件快照/备份；
3. 配置和密钥恢复准备；
4. 隔离环境恢复数据库；
5. 恢复对应文件；
6. 抽查产品、项目、决策和文件关联；
7. 记录RPO、RTO和问题。

验收：

- RPO ≤ 24小时；
- RTO ≤ 24小时；
- 被抽查文件SHA-256一致；
- 数据库记录与文件对象无不可解释缺失；
- 恢复过程不依赖个人记忆。

## 15. 覆盖率和门禁

覆盖率是最低防退化指标，不替代业务场景：

- 后端总体行覆盖率不低于80%；
- 状态机、权限策略和关键应用服务分支覆盖率不低于90%；
- 前端总体行覆盖率不低于70%；
- 关键表单、权限动作和错误处理模块不低于80%；
- 四条重大阶段门和两条端到端主链必须有E2E。

任何覆盖率豁免必须在合并说明中列出未覆盖风险和补充时间。

## 16. CI质量门禁

合并请求必须全部通过：

```text
后端格式 → 后端静态检查 → 类型检查
→ MySQL迁移检查 → 后端测试
→ 前端lint → vue-tsc → Vitest
→ OpenAPI契约 → 前端构建
→ Docker构建 → E2E冒烟
```

阻止合并：

- 任一失败测试；
- 未披露的跳过或xfail；
- 新增高严重度安全问题；
- 未生成迁移或迁移冲突；
- OpenAPI与实现不一致；
- 需求编号无测试或验收映射；
- 旧原型成为正式构建依赖。

Playwright允许一次重试仅用于收集诊断；首次失败后重试通过仍标记为不稳定，发布前必须处理。

## 17. 阶段质量门禁

### 阶段0

- 工程可重复启动；
- CI全部通过；
- 空库迁移成功；
- 正式构建不引用旧原型。

### 阶段1

- 默认拒绝权限有效；
- 审计失败回滚；
- Redis故障不丢发件箱事件；
- 文件补偿流程通过。

### 阶段2—5

- 对应PRD编号全部有测试证据；
- 纵向业务切片E2E通过；
- 权限、并发、失败和审计场景通过；
- 上一阶段回归测试无失败。

### 阶段6—7

- 钉钉和首批集成契约通过；
- 模拟迁移通过；
- 容量、安全、备份、恢复和回滚通过；
- 业务验收人签字确认。

## 18. 缺陷等级

| 等级 | 定义 | 发布处理 |
|---|---|---|
| P0 | 数据丢失、越权、决策错误、无法恢复 | 立即停止发布 |
| P1 | 主流程阻断、严重错误或无可靠规避 | 必须修复 |
| P2 | 次要流程错误且有明确规避 | 评估后决定 |
| P3 | 文案、样式或低影响体验问题 | 可进入后续迭代 |

权限绕过、历史版本覆盖、重复创建关键事实和审计缺失最低按P1处理。

## 19. 发布验收证据

每个候选发布必须保存：

- 提交号和版本；
- CI完整结果；
- 需求—测试追踪报告；
- 数据库迁移结果；
- E2E和业务验收结果；
- 安全和容量结果；
- 发布包SHA-256；
- 测试环境部署记录；
- 备份状态；
- 已知限制和回滚步骤。

任何未执行项必须明确写出，禁止以“默认通过”代替证据。
