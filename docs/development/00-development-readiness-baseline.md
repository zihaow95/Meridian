## 5. 首期交付边界

首期必须形成可运行的完整主干：

```text
提案 → 立案 → 立项 → 开发 → 上市 → 运营/退市
```

实现深度：

- 提案—立案—立项：完整业务规则和真实使用深度；
- 产品档案、任务、交付物、文件、权限、审计：完整核心模型；
- 开发—上市—运营/退市：流程真实可运行，专业模板先采用通用版本；
- 存量产品和在途项目：具备受控导入和确认链路；
- 钉钉：登录、通知和深链接；
- 外部专业系统：首期通过适配器、Excel/CSV或手工方式接入，不替代其业务职能。

## 6. 开发过程中不可改变的约束

- 产品机会资产、项目实例和产品资产保持独立；
- 产品—版本—SKU—渠道统一主模型不得分叉；
- 四个重大阶段门不得新增、删除或由模板绕过；
- 老品迭代必须挂在统一生命周期主干；
- 正式业务状态只能通过后端应用服务迁移；
- 权限必须采用RBAC + ABAC + 操作审计；
- 平台管理权与业务数据访问权分离；
- 文件、模板、产品版本、决策和数据快照不得覆盖历史；
- 项目当前组长是唯一A，原子任务只有一个个人R；
- Redis、Celery和查询汇总不作为业务事实来源；
- 旧原型不能成为正式代码依赖。

违反上述约束的实现必须先修订架构、PRD和TRD，不能通过代码变通。

## 7. 非阻塞业务配置

以下事项不阻止工程初始化，但必须在对应业务切片验收前确认：

| 配置 | 最迟完成时间 |
|---|---|
| 提案额度、联合成员上限、产品部评估字段 | 提案—立项切片验收前 |
| 重大阶段门决策摘要结构 | 首个重大阶段门联调前 |
| 角色—动作矩阵、数据敏感等级映射 | 权限业务联调前 |
| 默认阶段、任务、RACI和核心交付物模板 | 项目执行切片验收前 |
| 产品属性组、字典和字段顺序 | 产品档案切片验收前 |
| 首批指标、阈值、覆盖率和数据源优先级 | 运营监控切片验收前 |
| 存量产品、在途项目导入模板 | 迁移演练前 |
| 页面原型和详细交互 | 对应前端页面开发前 |
| 文件类型、大小、通知模板和保留周期 | 测试环境验收前 |
| 外部系统字段映射和同步频率 | 对应集成开发前 |

配置不得改变统一主干、权限默认拒绝原则或产品主模型。

## 8. 变更控制

### 8.1 基线内调整

以下内容可直接进入实施计划：

- 字段命名和索引优化；
- 不改变业务语义的API结构调整；
- 页面布局和交互优化；
- 模板、字典、阈值和通知配置；
- 测试和部署实现细节。

### 8.2 必须回到设计评审的变更

- 生命周期主干或重大阶段门变化；
- 产品、项目、机会关系变化；
- 产品—版本—SKU—渠道层级变化；
- 权限或数据主权原则变化；
- 历史版本不可变原则变化；
- 模块化单体转为多服务；
- RPO、RTO或部署基础设施发生实质变化。

设计变更必须记录原因、影响PRD/TRD编号、迁移影响和批准人。
