## 1. 文档目的

本文定义产品全生命周期治理平台的产品目标、用户范围、首期边界、全局业务规则、功能域划分和验收口径。各业务域的页面、流程、字段、异常和详细验收标准由子 PRD 展开；子 PRD 不得突破本文和架构设计确定的统一主干。

本文同时服务于：

- 管理层确认建设目标、治理边界和首期投入范围；
- 产品团队设计流程、页面和业务规则；
- 开发团队识别功能边界和依赖；
- 测试团队建立端到端验收场景。

## 2. 产品背景

公司现有产品管理过程分散在人员沟通、文档、文件和多个专业系统中，提案、立案、立项、开发、上市及运营决策之间缺少统一数字主线。管理层难以持续获取完整且必要的信息，业务部门也缺少清晰、透明、可追溯的责任与阶段门机制。

本系统不替代 ERP、MES、WMS、销售、财务和 BI 系统，而是在其上建立产品生命周期治理层，将产品机会、项目过程、产品档案、受控文件、关键决策和经营监控统一关联。

## 3. 产品目标

### 3.1 业务目标

1. 建立从提案到退市的统一生命周期主干。
2. 建立长期存在、持续完善的统一产品档案。
3. 让关键决策具备完整依据、责任和历史版本。
4. 让业务部门可以在明确授权下推进项目，减少非重大节点对管理层的依赖。
5. 让管理层通过一个入口掌握产品、项目、风险和经营议题。
6. 为后续数据分析和智能化应用建立可信、结构化、可追溯的数据基础。

### 3.2 首期成功标准

- 新品能够从提案完整运行到上市运营；
- 老品能够从经营风险信号运行到迭代生效或退市；
- 两条链路均可验证权限隔离、责任落实、文件版本、产品档案发布、阶段门决策和审计记录；
- 提案—立案—立项具备可实际投入使用的完整业务规则；
- 开发—上市—运营/退市具备可运行的通用流程，不以空壳页面代替；
- 存量产品和在途项目能够通过轻量迁移进入系统。

## 4. 非目标

首期不建设：

- 复杂可视化流程设计器；
- ERP、MES、WMS 等系统的深度双向业务替代；
- 自动处罚、自动冻结或复杂风险分级；
- 在线多人文档编辑；
- 完整多租户运营、计费和租户自助开通；
- 微服务体系；
- 由钉钉直接处理系统业务状态。

## 5. 用户与角色

| 用户/角色 | 主要诉求 |
|---|---|
| 老板/最终决策人 | 获取重大节点的充分依据并作出最终决策 |
| 经管会成员 | 评审机会、立项、首次上市和退市，监督经营议题 |
| 产品总监 | 管理产品组合、项目负责人、例外、变更和资源协调 |
| 产品经理/项目组长 | 推进项目、维护项目草稿、归口交付物和阶段门 |
| 提案负责人/联合成员 | 提交或参与提案，跟踪本人相关提案结果 |
| 部门负责人 | 分配本部门执行人，确认关键专业交付物 |
| 项目成员 | 执行任务、提交材料、参与协作 |
| 专业确认人 | 对指定版本的关键专业交付物确认或退回 |
| 经营监督人 | 维护经营数据、处理风险信号和经营议题 |
| 普通员工 | 查看授权范围内的公开产品和项目信息 |
| 系统管理员 | 管理账号、角色、模板、集成和平台运行，不默认查看敏感业务内容 |

## 7. 全局产品主干

```text
提案 → 立案 → 立项 → 开发 → 上市 → 运营/退市
```

### 7.1 立项前

提案、立案评估和立项候选统一沉淀为产品机会资产。产品机会资产是公司知识资产，但不是正式产品资产或项目实例。

### 7.2 立项通过

- 创建正式项目实例；
- 新品创建“研发中产品”及草稿初始版本；
- 老品迭代关联已有产品资产并创建产品变更记录；
- 产品机会资产与项目实例建立多对多来源关系；
- 初始化项目组、模板快照、任务、交付物和项目工作台。

### 7.3 项目完成

项目过程沉淀为产品履历。通过批准的产品版本、SKU 或渠道配置发布到当前有效产品档案，历史事实不覆盖。

## 8. 核心产品对象

| 对象 | 用户含义 |
|---|---|
| 产品机会资产 | 立项前的提案、评估、候选、暂缓、Pass 和复议记录 |
| 项目实例 | 立项通过后推动新品或老品变化的一次治理事件 |
| 产品资产 | 长期存在的正式产品主对象 |
| 产品版本 | 产品核心定义在某一时期的有效版本 |
| SKU | 售卖、库存和效期管理单元 |
| 渠道配置 | SKU 在具体渠道中的价格、状态、素材和监控配置 |
| 产品档案 | 面向用户统一呈现产品当前事实和历史履历的入口 |
| 任务 | 对具体执行工作的分派和跟踪 |
| 交付物 | 阶段门使用的正式证据 |
| 经营议题 | 由风险信号升级形成的轻量治理事项 |

## 9. 全局业务规则

### 9.1 阶段门状态

所有阶段门统一使用：

- 通过；
- 带例外通过；
- 待补充；
- 暂缓推进；
- Pass。

Pass 后原记录不可修改；复议形成新的版本记录。暂缓推进默认进入季度回看。

### 9.2 四个重大阶段门

1. 提案进入立案；
2. 立案进入立项；
3. 新品首次上市；
4. 产品退市。

重大阶段门记录经管会整体结论和老板最终决策，不记录个人投票。流程状态以老板最终决策为准，二者不一致时同时展示差异。

### 9.3 统一主干与场景化策略

新品、老品迭代、渠道定制、降本、包装变更和紧急变更必须使用统一主干。项目可以对具体阶段采用执行、复用、简化、豁免、不适用或并行处理，但必须按权限确认并留痕。

### 9.4 责任模型

- 部门责任模板定义主责、执行、协同和知会部门；
- 提案负责人、立案负责人、项目组组长分别是对应时期的唯一 A；
- 每个原子任务只设置一名个人 R；
- 需要多个部门执行时拆分为部门子任务；
- 专业确认人和阶段门处理人不混入 RACI；
- 部门参与不自动向部门全员授予敏感信息权限。

### 9.5 模板和项目快照

模板发布后形成不可变版本。项目创建时复制模板快照，后续模板调整只影响新项目。项目组可以在权限范围内延伸任务、交付物和计划时间，但不能直接删除核心必交付物。

### 9.6 产品档案发布

项目中的产品信息先作为草稿维护，经属性组确认和对应阶段门批准后发布到有效产品档案。未经批准，不得覆盖当前有效产品事实。

### 9.7 文件和决策

- 正式交付物必须上传受控版本；
- 阶段门引用具体文件版本和数据快照；
- 后续修订创建新版本，不覆盖历史；
- 专业确认绑定具体版本，新版本需要重新确认；
- 关键操作必须形成可检索审计记录。

### 9.8 权限

权限采用 RBAC + ABAC + 操作留痕，至少控制页面、字段、文件正文、下载、导出、分享、通知摘要和授权行为。平台管理权与业务数据访问权分离，关键角色由管理员人工配置，不根据钉钉组织身份自动授权。

### 9.9 数据主权

产品定义和生命周期治理数据以本系统为准；生产、库存、交易、财务和经营原始事实以对应专业系统为准。本系统保存来源、同步时间和决策快照，不直接覆盖外部权威数据。

## 10. 核心信息架构

### 10.1 生命周期看板

统一展示立项前的产品机会资产和立项后的项目实例，支持按阶段、状态、负责人和逾期情况查看。

### 10.2 项目工作台

集中处理阶段、任务、交付物、专业确认、文件版本、阶段门、例外、复议和审计。

### 10.3 产品档案

产品档案采用“固定核心字段 + 可配置属性组”，在保持统一产品—版本—SKU—渠道模型的前提下适配不同品类和特殊场景。

集中呈现：

- 产品基本信息；
- 产品版本和有效状态；
- SKU 和渠道配置；
- 营养成分和包装素材；
- 经营监控与风险信号；
- 来源提案、历史项目和迭代履历；
- 项目草稿与当前有效档案的差异。

### 10.4 我的待办

提供我的任务、我的交付物、待专业确认、待阶段门处理和逾期提醒。系统是唯一权威处理入口，钉钉只负责通知和跳转。

## 12. 迁移需求

### 12.1 存量产品

- 创建来源为“存量产品导入”的产品资产和导入基线版本；
- 支持表单和 Excel 维护；
- 基线允许部分完整，不阻止收录；
- 产品总监一次确认后发布；
- 基线发布后的有效事实变化必须走产品迭代项目。

### 12.2 在途项目

- 新项目从系统启用日起必须走完整流程；
- 在途项目按实际当前阶段导入；
- 迁移基线记录阶段、负责人、项目组、已有交付物、历史决策摘要和迁移日期；
- 历史资料作为迁移历史资料，不补造系统阶段门；
- 产品总监确认后从当前阶段继续运行；
- 无继续治理价值的临近结束项目可以只归档。

## 14. 首期验收

### 14.1 新品场景

从正式提案开始，完成立案、立项、项目创建、开发任务、交付物、专业确认、首次上市决策、产品档案发布和运营监控。

### 14.2 老品场景

从经营数据产生风险信号，经经营议题研判后创建产品迭代项目，完成草稿变更、批准、生效并保留前后版本；或完成退市决策和状态变更。

### 14.3 共通验收

- 未授权用户无法查看敏感内容；
- 部门模板能够落实到具体个人任务；
- 所有关键决策能够追溯到当时的文件版本和数据快照；
- 钉钉通知不能直接改变业务状态；
- 模板更新不影响已创建项目；
- 存量产品和在途项目可以按迁移规则进入系统。

## 15. 后续配置项

以下内容不改变总体产品架构，在子 PRD 或实施配置阶段确定：

- 各阶段核心交付物及专业模板；
- 产品部评估字段；
- 决策摘要展示结构；
- 产品档案属性组和字段顺序；
- 首批经营指标、阈值和数据映射；
- 数据敏感等级与字段映射；
- 具体角色权限矩阵；
- 提案额度、联合成员人数等业务参数；
- 页面原型及详细交互；
- 分阶段发布计划。

配置不得创建脱离统一主干的独立流程或子系统。
