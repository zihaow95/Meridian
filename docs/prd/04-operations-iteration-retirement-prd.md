# 子 PRD 04：运营监控、迭代与退市

版本：V0.1  
日期：2026-06-29  
状态：已确认基线

基线确认日期：2026-07-02
上游依据：总体 PRD、产品全生命周期架构设计说明

## 1. 目标与范围

本模块负责产品上市后的经营监控、风险信号、经营议题、产品迭代触发和退市闭环。

核心原则：

- 底层按 SKU × 渠道 × 时间采集；
- 上层按 SKU 和产品汇总；
- 系统只产生风险信号，不自动启动迭代；
- 经营监督人负责首次研判；
- 产品迭代仍进入统一提案—立案—立项主干；
- 重大决策最终由经管会和老板处理。

首期保持轻量，不建设复杂风险等级、自动处罚或自动冻结。

## 2. 用户与职责

| 角色 | 职责 |
|---|---|
| 经营监督人 | 维护数据、查看风险信号、首次研判、关闭轻微议题 |
| 产品经理/产品总监 | 制定产品优化或迭代方案 |
| 经管会 | 研判重大经营议题、形成迭代或退市整体结论 |
| 老板/最终决策人 | 对产品退市作出最终决策 |
| 销售/市场/供应链/财务 | 提供专业数据和研判输入 |
| 系统管理员 | 配置数据源、指标和同步任务，不默认查看敏感业务值 |

经营监督人仅因该角色获得经营监控和议题权限，不自动获得成本、供应商、工艺或立项全文权限。

### 2.1 关键用户故事

- 作为经营监督人，我可以看到有依据的风险信号并快速关闭轻微问题或升级议题。
- 作为产品经理，我可以从经营议题获得完整问题背景并形成产品迭代方案。
- 作为经管会成员，我可以基于产品汇总和 SKU/渠道明细研判是否推进迭代。
- 作为老板，我可以基于锁定数据快照作出退市最终决策。
- 作为数据维护人员，我可以导入、修正数据，同时保留原始来源和处理痕迹。

### 2.2 通用前置条件

- 产品、有效版本、SKU 和渠道配置已存在；
- 数据源、字段映射、指标口径和阈值已发布；
- 经营监督人及其产品/渠道范围已配置；
- 数据批次必须通过结构校验后才能参与汇总；
- 口径或数据不足时系统不得生成伪正常结论。

## 3. 数据模型

### 3.1 数据粒度

经营事实数据最小标准粒度为：

```text
SKU × 渠道 × 时间
```

时间粒度支持日、周、月和季度。原始数据保留其实际粒度，汇总时不得伪造更细数据。

### 3.2 数据层次

```text
数据源
→ 接入批次
→ 暂存与校验
→ 标准化映射
→ SKU/渠道经营事实
→ SKU 汇总
→ 产品汇总
→ 风险信号
```

### 3.3 数据来源

首期同时支持：

- 自动接口同步；
- Excel/CSV 导入；
- 手动录入和修正。

每条数据至少记录来源、批次、外部标识、业务期间、导入/同步时间和当前有效状态。

## 4. 数据接入和校验

### 4.1 接口同步

- 同步失败不能覆盖上次有效数据；
- 记录成功、部分成功或失败批次；
- 展示失败原因和重试状态；
- 重复批次按数据源和业务键识别，不重复累计。

### 4.2 Excel/CSV 导入

- 提供标准模板；
- 上传后先校验再入库；
- 展示新增、更新、跳过和错误数量；
- 错误行提供具体原因；
- 用户确认后才写入经营事实。

### 4.3 校验规则

至少校验：

- SKU 和渠道映射存在；
- 时间范围合法；
- 数值字段类型正确；
- 同一业务键是否重复；
- 单位和币种是否匹配；
- 数据是否超出配置的合理范围。

合理范围异常默认警告，不自动拒绝；结构性错误必须阻止写入。

## 5. 人工确认有效值

当接口值需要人工修正时：

- 经营监督人可在权限范围内创建“人工确认有效值”；
- 原接口数据不删除、不覆盖；
- 看板、汇总和风险信号使用当前有效值；
- 页面明确标记人工确认；
- 记录修改人、时间、原值、新值和适用期间；
- 有效值持续到再次修改或撤销；
- 不建设额外双人审批。

人工确认只改变本系统经营判断使用值，不改变外部专业系统作为原始事实权威来源的地位。

## 6. 指标和汇总

### 6.1 首期指标能力

首期支持配置：

- 指标名称和说明；
- 计算口径；
- 来源字段；
- 汇总方式；
- 时间窗口；
- 适用产品/SKU/渠道；
- 预警阈值；
- 数据更新时间。

首批具体指标由运营配置确定，不写死在程序中。

### 6.2 推荐基础指标

- 销量/销售额；
- 动销和补货；
- 毛利和费用率；
- 库存和库存周转；
- 临期和退货；
- 缺货；
- 复购、退款和客诉；
- 用户净流入/净流出；
- 竞品或市场份额输入。

### 6.3 汇总原则

- 产品汇总必须可下钻至 SKU 和渠道；
- 汇总页面显示数据覆盖范围和更新时间；
- 不同渠道口径不一致时必须标记，不强行相加；
- 人工有效值参与汇总并显示人工标记；
- 决策引用时生成不可变数据快照。

## 7. 风险信号

### 7.1 风险来源

系统可根据配置生成：

- 销售下滑信号；
- 用户净流出信号；
- 竞品抢占信号；
- 库存、临期或退货异常；
- 毛利或费用异常；
- 连续一个季度无法消化“四分之一效期最低生产量”；
- 其他配置规则。

“四分之一效期最低生产量”的具体计算参数由业务配置维护，信号必须显示公式、数据期间、阈值和实际值。

### 7.2 信号状态

```text
新产生 → 已查看
       ├─ 关闭
       └─ 升级为经营议题
```

风险信号不是审批单，不设置复杂风险等级。相同产品、指标和期间的重复信号应合并提示或关联，避免通知轰炸。

### 7.3 关闭

- 经营监督人可以直接关闭轻微信号；
- 关闭必须选择或填写简短理由；
- 关闭不删除信号和数据依据；
- 后续新周期再次触发时可以重新产生信号。

## 8. 经营议题

### 8.1 创建

经营监督人确认风险需要业务研判时，将一个或多个信号升级为经营议题。议题至少包含：

- 标题；
- 关联产品、SKU 和渠道；
- 触发信号；
- 数据快照；
- 现象摘要；
- 责任人；
- 建议研判时间。

### 8.2 议题状态

```text
待研判 → 研判中
         ├─ 继续观察
         ├─ 采取轻量经营动作
         ├─ 转产品迭代提案
         ├─ 提交退市评审
         └─ 关闭
```

状态保持轻量，不增加风险分级审批链。

### 8.3 研判结论

支持：

- 继续观察；
- 调价建议；
- 渠道调整；
- 市场/销售动作；
- 供应或成本优化；
- 产品迭代；
- 暂停销售；
- 退市；
- 关闭。

调价、渠道调整或其他动作是否需要产品迭代项目，由其是否改变有效产品事实和现有权限规则判断。

## 9. 产品迭代触发

### 9.1 转换原则

系统不直接从信号创建项目。转换链路为：

```text
风险信号 → 经营议题 → 产品迭代提案
→ 立案 → 立项 → 产品迭代项目
```

经营议题转换时预填：

- 产品和版本；
- 触发信号及数据快照；
- 问题摘要；
- 建议方向；
- 关联 SKU/渠道；
- 原议题责任人。

提案负责人仍须具备提案资格；经管会决定是否立案和立项。

### 9.2 迭代类型

前台统一称“产品迭代”，可覆盖：

- 产品升级；
- 配方/口味优化；
- 降本；
- 包装变更；
- 渠道定制；
- 调价；
- 供应调整；
- 暂停销售；
- 退市准备。

底层按产品变更记录具体影响范围。

### 9.3 生效

- 普通迭代生效由产品总监确认；
- 涉及退市时进入固定重大阶段门；
- 批准后按子 PRD 02 发布产品版本、SKU 或渠道配置；
- 经营议题关联项目并跟踪最终结果。

## 10. 退市

### 10.1 发起

退市可以来自：

- 经营议题；
- 产品组合复盘；
- 质量/合规风险；
- 战略调整；
- 产品总监或经管会直接提出。

### 10.2 退市评审材料

至少包含：

- 产品、版本、SKU 和渠道范围；
- 销售、毛利、库存、临期和客诉摘要；
- 供应、合同和渠道影响；
- 库存处理方案；
- 客户和市场沟通方案；
- 停产、停售和系统状态计划；
- 替代产品或用户承接方案；
- 决策数据快照和文件版本。

### 10.3 重大阶段门

产品退市是固定重大阶段门：

- 经管会形成整体结论；
- 老板作出最终决策；
- 流程以老板决策为准；
- 决策绑定退市范围、计划和数据快照。

### 10.4 执行

退市批准后按计划：

- 更新产品、版本、SKU 和渠道状态；
- 设置停产、停售或退市日期；
- 保留历史档案和经营数据；
- 继续跟踪剩余库存、临期和售后事项；
- 完成后关闭退市项目/议题。

退市不是物理删除。

## 11. 看板和页面

### 11.1 产品经营看板

展示产品汇总并支持下钻：

- 当前状态；
- 核心指标和趋势；
- SKU/渠道贡献；
- 数据更新时间和覆盖率；
- 风险信号；
- 经营议题；
- 关联迭代项目。

### 11.2 风险信号中心

支持按产品、指标、时间、状态和责任人筛选，查看触发公式、阈值、实际值、来源数据和处理历史。

### 11.3 经营议题工作台

展示信号依据、数据快照、研判记录、行动、关联提案/项目和最终结果。

### 11.4 产品档案运营页

在产品档案内展示经营摘要、有效版本、风险、议题、迭代和退市履历。

## 12. 通知

至少通知：

- 新风险信号；
- 信号连续未查看提醒；
- 经营议题创建、责任人变更和研判截止；
- 议题转换为迭代提案；
- 迭代或退市决策结果；
- 数据同步/导入失败。

首期不因提醒超期自动改变状态或升级审批。

## 13. 权限

- 经营监督人可查看经营监控数据、修正值、信号和议题；
- 经营监督人可关闭轻微信号，无需额外复核；
- 成本、供应商、工艺等敏感信息仍按其他角色授权；
- 产品经理只查看其负责产品或项目的受控经营信息；
- 经管会和老板按重大议题获得决策材料；
- 系统管理员可维护数据源和规则，但默认不能读取敏感业务内容；
- 导出经营明细需要独立权限并留痕。

## 14. 异常与边界

- 某周期数据缺失时不生成“正常”结论，显示数据不足；
- 迟到数据导致指标重算时保留原信号和重算记录；
- 指标口径修改只影响新计算周期，历史决策快照不重写；
- SKU 或渠道映射失效时数据进入待处理区，不归入错误产品；
- 人工确认值撤销后恢复使用当前有效来源值并重算后续信号；
- 一个议题可关联多个信号，一个信号只能有一个当前主议题；
- 已转迭代提案的议题不能直接删除。

## 16. 验收场景

### 16.1 接口和人工值

接口同步一条 SKU/渠道月度数据，经营监督人创建人工确认有效值；看板和信号使用人工值，同时保留接口原值和来源。

### 16.2 风险信号

产品连续一个季度未达到配置的“四分之一效期最低生产量”消化阈值，系统生成信号并展示公式、阈值、实际值和数据期间，但不自动创建迭代项目。

### 16.3 轻微信号关闭

经营监督人填写简短理由直接关闭信号，无需额外审批；下一周期再次触发时产生新信号。

### 16.4 转产品迭代

经营议题研判为产品优化，系统创建预填的迭代提案；提案仍经过立案和立项重大阶段门，通过后关联正式迭代项目。

### 16.5 退市

退市材料引用当期经营快照，经管会和老板完成重大阶段门后，产品及适用 SKU/渠道按计划停用，历史档案和剩余库存跟踪仍可访问。
