# 子 PRD 03：开发—上市项目执行

版本：V0.1  
日期：2026-06-29  
状态：已确认基线

基线确认日期：2026-07-02
上游依据：总体 PRD、产品全生命周期架构设计说明

## 1. 目标与范围

本模块负责立项通过后的项目执行，包括项目初始化、阶段计划、任务、交付物、专业确认、试销验证、上市准备、首次上市决策和产品档案发布。

首期目标不是固化一套不可变化的专业流程，而是在统一主干上提供真实可运行的默认模板。新品完整执行，老品迭代可按授权复用、简化、豁免、不适用或并行处理。

## 2. 用户与职责

| 角色 | 职责 |
|---|---|
| 产品总监 | 任命/更换项目组长、确认例外和重要调整、协调资源 |
| 项目组长 | 项目唯一 A，负责整体计划、交付物归口和阶段门提交 |
| 项目副组长 | 协助推进和跨部门沟通，不形成第二个 A |
| 部门负责人 | 为部门任务指定 R，确认关键专业交付物 |
| 项目成员 | 执行任务和编制交付物 |
| 专业确认人 | 对具体受控版本确认或退回 |
| 经管会 | 对新品首次上市形成整体结论 |
| 老板/最终决策人 | 对新品首次上市作出最终决策 |

### 2.1 关键用户故事

- 作为项目组长，我可以从模板快照获得完整项目骨架，并对项目整体进度负责。
- 作为部门负责人，我可以把部门任务落实到具体个人并处理人员变化。
- 作为项目成员，我可以清楚看到本人任务、截止时间、交付物和阻塞项。
- 作为专业确认人，我可以对指定交付物版本确认或退回。
- 作为产品总监，我可以确认简化、复用、豁免、重要调整和普通迭代生效。
- 作为经管会或老板，我可以基于锁定的首次上市材料完成重大决策。

### 2.2 通用前置条件

- 立项重大阶段门已经通过并成功创建项目实例；
- 项目组长、项目类型和关联产品/机会已确定；
- 有效项目模板、部门责任模板和权限策略已发布；
- 文件、审计、待办和产品草稿服务可用；
- 初始化依赖缺失时必须整体失败，不能创建半初始化项目。

## 3. 项目初始化

### 3.1 触发

立项通过后由系统自动初始化项目，不允许用户手工重复创建同一立项项目。

### 3.2 初始化内容

- 项目编号和名称；
- 项目类型：产品项目、产品迭代项目；
- 来源机会资产；
- 关联产品和变更范围；
- 项目组长、副组长和成员；
- 当前阶段；
- 项目模板快照；
- 阶段、任务和交付物；
- 默认计划和里程碑；
- 产品档案草稿；
- 权限和通知范围。

### 3.3 模板快照

- 项目创建时复制已发布模板版本；
- 后续模板修改不影响既有项目；
- 项目组长和产品总监可在快照基础上延伸任务、推荐交付物和计划；
- 核心必交付物不得直接删除，只能申请豁免；
- 所有项目级调整记录操作人、时间和前后值。

## 4. 默认阶段模板

首期开发—上市默认模板如下。具体事项、周期和交付物可按品类配置，但不能脱离开发和上市主干。

| 阶段 | 目标 | 默认关键结果 |
|---|---|---|
| D1 项目启动与产品定义 | 确认范围、计划、草稿档案和开发基线 | 项目计划、产品定义草稿 |
| D2 配方打样与体验验证 | 形成可评价样品和标准样 | 样品记录、品评/测试报告 |
| D3 工艺放大与质量验证 | 验证试产、稳定性、保质期和量产条件 | 试产、稳定性、量产准入资料 |
| D4 工程化与试销准备 | 准备包装、系统建档、渠道、资源和指标 | 试销计划、数据口径、备货方案 |
| D5 上市验证/试销 | 验证产品力、渠道适配和经营模型 | 试销报告和分支决策 |
| L1 正式上市方案 | 完成市场、产品、供应、渠道和预算准备 | 上市方案、上市版本、资源计划 |
| L2 首次上市阶段门 | 对新品首次上市作出重大决策 | 经管会结论、老板决策 |
| L3 发布与运营交接 | 发布产品档案并进入运营监控 | 有效档案、运营责任和监控配置 |

### 4.1 D1 项目启动与产品定义

默认任务：

- 确认项目目标、范围、成功标准和不做事项；
- 确认项目组和部门责任；
- 设置计划、里程碑和风险；
- 建立产品档案草稿；
- 完善产品规格、目标成本、用户、卖点和建议零售价；
- 明确拟用生产方、工厂、合作模式和技术路线。

默认交付物：

- 项目计划；
- 产品开发任务书；
- 产品规格草稿；
- 目标成本和毛利测算；
- 风险清单。

### 4.2 D2 配方打样与体验验证

默认任务：

- 配方和工艺开发；
- 样品版本记录；
- 内部品评；
- 必要的消费者或渠道测试；
- 标准样确认；
- 更新产品定义、配方和营养草稿。

默认交付物：

- 样品版本记录；
- 内部品评报告；
- 消费者/渠道测试报告；
- 标准样确认单。

### 4.3 D3 工艺放大与质量验证

默认任务：

- 组织试产；
- 验证工艺参数和生产损耗；
- 验证稳定性、保质期、微生物和感官衰减；
- 校准制造成本；
- 验证仓储、物流和追溯要求；
- 形成量产准入结论。

默认交付物：

- 试产报告；
- 稳定性/保质期验证报告；
- 量产成本修正；
- 量产准入评审表。

### 4.4 D4 工程化与试销准备

该阶段用于试销准备，不等同正式上市准备。

默认任务：

- 包装与卖点表达；
- 拟定产品/SKU/渠道编码并绑定外部建档状态；
- 准备试销备货和供应方案；
- 选择试销渠道；
- 配置指标、数据口径和采集方式；
- 准备渠道、培训和测试资源。

默认交付物：

- 包装与卖点表达确认单；
- 系统建档检查单；
- 试销测试计划；
- 渠道与资源计划；
- 指标与数据口径表。

### 4.5 D5 上市验证/试销

默认支持以下试销任务，项目可按批准策略执行、并行、简化或不适用：

| 轮次 | 主要目的 | 建议周期 |
|---|---|---|
| A 轮 | 小范围验证基础可售性和质量风险 | 1–2 周 |
| B 轮 | 验证线上表达、转化和复购 | 2–3 周 |
| C 轮 | 验证线下铺货、陈列和动销 | 2–4 周 |
| D 轮 | 验证跨渠道放量和供应响应 | 3–4 周 |

B、C 轮可并行。每轮必须记录适用渠道、样本/范围、时间、指标口径、结果和结论，不强制所有项目执行全部四轮。

试销分支：

- 表现不佳：终止或归档，停止新增资源；
- 表现一般：返回 D2、D3 或 D4 优化；
- 表现良好：进入正式上市方案；
- 表现优秀：进入正式上市方案并加快资源确认。

### 4.6 L1 正式上市方案

默认任务：

- 正式上市分支确认；
- 市场营销方案；
- 产品微调和上市版本确认；
- 供应链与渠道资源协同；
- 上市预算与资源投入；
- 上市风险和回退方案；
- 首批运营监控指标配置。

默认交付物：

- 正式上市市场营销方案；
- 产品上市版本确认单；
- 供应链与渠道资源协同方案；
- 上市预算与资源投入审批表；
- 正式上市检查表。

### 4.7 L2 首次上市阶段门

新品首次上市是固定重大阶段门：

- 经管会形成整体结论；
- 老板作出最终决策；
- 决策绑定上市方案、产品版本、预算、资源和风险快照；
- 不记录个人投票；
- 最终状态以老板决策为准。

结果支持通过、带例外通过、待补充、暂缓推进和 Pass。重大阶段门的例外说明不能替代老板最终决策。

### 4.8 L3 发布与运营交接

上市通过后：

- 发布批准的产品版本、SKU 和渠道配置；
- 锁定上市交付物和决策依据；
- 设置实际上市时间；
- 创建运营监控范围；
- 指定运营责任人/经营监督人；
- 项目进入运营阶段；
- 发送上市结果通知。

## 5. 老品迭代执行策略

老品迭代仍保留统一主干和阶段记录。项目组长可以提出：

- 复用：绑定历史交付物和有效依据；
- 简化：减少任务或降低模板深度；
- 豁免：不执行原本要求的阶段或交付物；
- 不适用：由项目类型策略预设；
- 并行处理：允许阶段或任务并行。

产品总监确认规则：

- 复用、简化、并行由产品总监确认；
- 豁免由产品总监直接确认或升级经管会；
- 不适用由项目类型模板带出，项目内不能随意改写；
- 阶段门通过时可以一并确认例外；
- 统一记录为“例外通过说明”，不建设复杂例外审批子系统。

普通老品变更生效由产品总监批准；产品退市仍进入重大阶段门。

## 6. 任务模型

### 6.1 任务字段

每个任务至少包含：

- 名称和说明；
- 所属阶段；
- 部门责任类型；
- 唯一个人 R；
- 计划开始/截止时间；
- 状态；
- 关联交付物；
- 前置任务；
- 逾期标记；
- 创建来源：模板或项目自定义。

### 6.2 任务状态

```text
未开始 → 进行中 → 已完成
        ├─ 阻塞
        └─ 已取消
```

已取消仅适用于非核心自定义任务，并要求留痕。核心任务通过阶段处理策略决定，不直接取消。

### 6.3 部门到个人

- 部门模板决定主责、执行、协同和知会部门；
- 部门负责人从执行部门指定个人 R；
- 项目组长保持项目整体唯一 A；
- 多部门执行必须拆分为原子任务；
- C、I 默认按部门发送协同或结论通知，关键事项可指定个人。

既有部门 RACI 资料作为模板迁移来源时，A、R、A/R、C/R 分别转换为主责、执行、主责兼执行和协同执行；转换后仍须按本 PRD 的唯一项目 A 和唯一个人 R 规则校正。

## 7. 交付物和专业确认

### 7.1 三层交付物

| 类型 | 规则 |
|---|---|
| 核心必交付物 | 不能删除，只能申请豁免 |
| 模板推荐交付物 | 可由项目组调整 |
| 项目自定义交付物 | 项目组按实际新增 |

### 7.2 受控版本

- 上传形成草稿版本；
- 提交评审后锁定；
- 阶段门引用具体版本；
- 通过后成为受控版本；
- 修订生成新版本；
- 作废不物理删除。

### 7.3 专业确认

- 只对关键专业交付物启用；
- 默认由部门负责人本人确认，产品总监可指定；
- 产品总监和项目组长可调整确认人并留痕；
- 确认动作仅为通过或退回修改；
- 确认绑定具体版本；
- 新版本被阶段门引用前必须重新确认。

## 8. 阶段门提交和处理

提交阶段门前系统检查：

- 核心任务状态；
- 核心交付物及豁免；
- 专业确认；
- 产品档案草稿完整性；
- 计划和风险；
- 引用文件版本。

非重大阶段门由配置的业务负责人或产品总监处理。结果统一为通过、带例外通过、待补充、暂缓推进和 Pass。

待补充返回当前阶段；暂缓进入候选/暂缓池；Pass 保留复议能力。

## 9. 计划调整、逾期和资源冲突

### 9.1 调整权限

| 调整 | 权限 |
|---|---|
| 不影响交付物、阶段门、跨部门资源和关键节点的轻微调整 | 项目组长直接调整并留痕 |
| 跨部门任务、关键里程碑、专业确认或阶段门调整 | 产品总监确认 |
| 产品总监无法协调的资源冲突 | 升级经管会 |

### 9.2 逾期

首期只做提醒和看板标红：

- 任务逾期提醒 R 和项目组长；
- 交付物逾期提醒项目组长；
- 专业确认逾期提醒确认人和项目组长；
- 阶段门逾期提醒项目组长和产品总监。

首期不自动冻结、不自动改变项目状态、不自动处罚。

### 9.3 先执行、后补确认

- 仅总监级人员可以发起；
- 必须绑定项目、事项和待补内容；
- 系统持续提醒并标红；
- 超期未补时升级提示经管会；
- 系统不自动回滚现实操作。

## 10. 在途项目迁移

系统启用时的在途项目按当前实际阶段导入：

- 记录当前阶段、负责人、项目组和计划；
- 导入已有任务、交付物和历史文件；
- 记录历史决策摘要和迁移日期；
- 历史文件标记为迁移历史资料；
- 不补造系统阶段门和专业确认；
- 产品总监确认迁移基线后继续运行；
- 无继续治理价值的临近结束项目只归档。

## 11. 页面需求

### 11.1 生命周期看板

展示项目阶段、负责人、阶段门、关键交付物、逾期、暂缓和例外状态，支持按产品、项目类型、阶段、负责人和状态筛选。

### 11.2 项目工作台

至少包含：

- 项目概览；
- 阶段和里程碑；
- 任务；
- 交付物；
- 产品档案草稿；
- 专业确认；
- 阶段门；
- 风险与例外；
- 文件和审计。

### 11.3 我的待办

统一展示任务、交付物、专业确认和阶段门待办。完成动作必须在系统内执行。

## 12. 权限

- 项目组长可维护计划、分配成员和提交阶段门；
- 任务 R 只能编辑本人任务及获授权草稿；
- 部门负责人可分派本部门 R；
- 专业确认人只能处理指定交付物版本；
- 产品总监可处理例外、重要调整和普通迭代生效；
- 经管会和老板只获得对应重大决策权限及授权材料；
- 项目成员权限随项目身份和敏感等级计算，不因部门参与自动扩大。

### 12.1 异常与边界

- 项目初始化重复触发时返回既有项目，不重复生成任务和产品对象；
- 任务 R 离职或停用后进入待重新分派列表，历史完成记录不改写；
- 阶段门提交期间交付物产生新版本时，本次提交继续引用已锁定版本；
- 并行任务修改同一产品草稿字段时提示冲突，不静默覆盖；
- 阶段处理策略改变后，系统重新计算阻塞项但保留原确认记录；
- 产品档案发布失败时项目阶段门结果保留，项目显示“待发布修复”，有效档案不部分更新；
- 审计写入失败时负责人变更、阶段门和例外操作必须失败。

| EXE-014 | 支持在途项目当前阶段迁移 | P0 |

## 14. 验收场景

### 14.1 新品完整执行

立项通过后自动初始化项目；项目依次完成打样、试产、试销、上市方案和首次上市重大阶段门；批准后产品档案发布并进入运营。

### 14.2 老品包装迭代

项目组提出复用历史质量资料、简化试销并跳过不适用任务；产品总监确认后项目完成包装草稿、专业确认和变更生效，旧包装版本保留。

### 14.3 文件修订

专业确认通过后交付物生成新版本，原确认不自动沿用；新版本必须重新确认才能提交阶段门。

### 14.4 逾期和调整

任务逾期只提醒和标红，不自动改变项目状态；项目组长可直接调整轻微任务，跨部门里程碑调整必须由产品总监确认。

### 14.5 在途项目

现有项目从 D3 阶段迁移，历史资料作为迁移文件保留，产品总监确认基线后继续执行，不产生伪造的 D1/D2 阶段门。
