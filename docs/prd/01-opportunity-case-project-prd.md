# 子 PRD 01：提案—立案—立项

版本：V0.1  
日期：2026-06-29  
状态：已确认基线

基线确认日期：2026-07-02
上游依据：总体 PRD、产品全生命周期架构设计说明

## 1. 目标与范围

本模块负责把员工产品想法转化为可评审的产品机会资产，并通过提案、立案和立项两个重大阶段门形成正式项目。

首期范围包括：

- 提案资格、联合提案和季度额度；
- 提案草稿、提交、评审和可见性；
- 立案负责人、副组长和产品部评估；
- 立项决策、项目创建和产品资产创建；
- 待补充、暂缓推进、Pass、复议；
- 多提案合并立项和单提案拆分立项；
- 候选机会池、季度回看和历史追溯。

不包括立项后的开发任务执行，相关需求见子 PRD 03。

## 2. 用户与职责

| 角色 | 职责 |
|---|---|
| 提案负责人 | 创建、完善和正式提交提案，对提案质量负责 |
| 联合提案成员 | 提供想法、信息和协作内容，不独立提交 |
| 产品经理 | 可独立提案；可被任命为立案负责人 |
| 销售/市场部门负责人 | 组织部门提案并作为正式提案负责人 |
| 产品总监 | 任命立案负责人、处理例外、合并拆分和复议退回阶段 |
| 经管会 | 对进入立案、进入立项形成整体结论 |
| 老板/最终决策人 | 对进入立案、进入立项作出最终决策 |
| 普通员工 | 通过加入合格提案组参与提案 |
| 系统管理员 | 配置提案资格角色、额度和通知，不参与业务决策 |

### 2.1 关键用户故事

- 作为具备资格的提案负责人，我可以组织成员、完善提案并正式提交，以进入公司评审。
- 作为普通员工，我可以加入合格提案组贡献想法，但不能绕过资格独立提交。
- 作为产品总监，我可以任命最适配的立案负责人，并处理合并、拆分、例外和人员更换。
- 作为经管会成员或老板，我可以基于固定版本材料形成整体结论或最终决策。
- 作为提案参与者，我可以持续追踪本人相关提案的待补充、暂缓、Pass 和复议结果。

### 2.2 通用前置条件

- 用户、部门和在职状态已同步或维护；
- 提案资格、额度归属和重大决策角色已人工配置；
- 提案、立案模板和敏感等级已发布；
- 文件存储、审计和站内待办可用；
- 任一关键前置配置缺失时，系统应阻止对应提交并明确指出缺失项。

## 3. 核心对象和状态

### 3.1 产品机会资产

首次保存提案草稿时创建产品机会资产。该对象持续承载提案、立案、立项候选、暂缓、Pass 和复议历史，不因后续创建项目而删除。

### 3.2 提案状态

```text
草稿 → 已提交 → 评审中
                 ├─ 待补充 → 重新提交
                 ├─ 暂缓推进 → 候选机会池
                 ├─ Pass → 可发起复议
                 └─ 通过 → 立案中
```

### 3.3 立案状态

```text
待任命负责人 → 评估中 → 待立项评审
                         ├─ 待补充 → 补充评估
                         ├─ 暂缓推进 → 候选机会池
                         ├─ Pass → 可发起复议
                         └─ 通过 → 创建正式项目
```

状态变化必须记录操作人、时间、来源阶段、目标状态和绑定的评审版本。

## 4. 提案资格和联合提案

### 4.1 正式提交资格

- 产品经理可以作为个人提案负责人；
- 销售端口和市场营销部门由配置的部门负责人作为部门提案负责人；
- 普通员工不能独立提交正式提案，只能作为联合成员参与；
- 系统只允许拥有有效提案资格的在职用户执行“正式提交”；
- 提案资格由系统管理员人工配置，不根据钉钉岗位自动授予。

### 4.2 联合提案

- 提案负责人可以邀请普通员工或其他授权人员加入提案组；
- 被邀请人接受后才能查看和协作编辑该提案；
- 联合成员不得把本人替换为正式负责人以绕过提案资格；
- 提案提交后，成员增删由提案负责人发起并留痕；
- 成员人数上限作为后台业务参数配置；
- 非产品经理来源提案进入立案后，必须从原提案组指定一名副组长。

### 4.3 提案归属与额度

- 产品经理提案计入该产品经理个人季度额度；
- 部门提案计入对应部门季度额度；
- 联合成员只记录贡献，不重复计数；
- 一个提案只允许一个额度归属；
- 季度额度不足不阻止保存草稿，但提交时按配置决定“阻止提交”或“显示缺口提醒”；首期默认只提醒，不自动补造提案；
- 额度统计必须区分草稿、有效提交、撤回和作废记录，默认仅有效提交计数。

首期默认最低额度：

| 归属 | 每季度最低有效提案数 |
|---|---:|
| 每名产品经理 | 3 |
| 平台电商销售部门 | 3 |
| 内容电商销售部门 | 3 |
| 线下销售部门 | 3 |
| 市场营销部门 | 3 |

管理员可以按季度调整数值和对应组织，但调整不得改写已结束季度的统计结果。

## 5. 提案创建和提交

### 5.1 最小必填内容

正式提案至少包含：

1. 市场分析；
2. 核心卖点；
3. 消费人群/用户需求；
4. 建议零售价。

同时记录：

- 提案标题；
- 提案类型初判：新品、老品迭代候选或暂不确定；
- 提案负责人和联合成员；
- 归属部门；
- 公开摘要；
- 参考产品、竞品或已有产品关联；
- 附件和来源说明。

普通员工提供的灵感可由合格提案负责人纳入联合提案，但系统不单独建立“匿名投稿”旁路。

### 5.2 草稿规则

- 草稿仅提案组和授权人员可见；
- 草稿可以多次保存；
- 系统记录最后编辑人和时间；
- 同一负责人可以同时维护多个草稿；
- 草稿删除采用逻辑删除，提交后不得物理删除。

### 5.3 提交校验

提交时系统必须校验：

- 提案负责人具备当前有效资格；
- 四项最小内容完整；
- 公开摘要已填写；
- 额度归属唯一；
- 联合成员均为有效用户；
- 当前不存在未完成的重复提交操作。

校验失败时保留草稿并逐项提示，不改变业务状态。

### 5.4 撤回

- 进入正式评审前，提案负责人可以撤回并回到草稿；
- 已进入评审后不能自行撤回，由产品总监退回；
- 撤回和退回均保留版本和原因。

## 6. 提案评审：进入立案

### 6.1 评审材料

经管会评审页面至少展示：

- 提案四项核心内容；
- 提案组和额度归属；
- 公开摘要；
- 关联竞品或已有产品；
- 附件清单；
- 当前材料版本；
- 与已有机会、项目和产品的相似关联提示。

### 6.2 重大阶段门

“提案进入立案”是固定重大阶段门：

- 记录经管会整体结论；
- 记录老板最终决策；
- 不记录个人投票；
- 流程状态以老板最终决策为准；
- 二者不一致时同时展示差异；
- 决策必须绑定当时提案版本和附件版本。

### 6.3 评审结果

| 结果 | 系统动作 |
|---|---|
| 通过 | 进入立案，等待任命立案负责人 |
| 待补充 | 返回提案负责人补充，保留原评审版本 |
| 暂缓推进 | 进入候选机会池，默认季度回看 |
| Pass | 锁定本次评审版本，允许未来复议 |

“暂缓推进”至少填写暂缓原因或重启触发条件，允许只填写重启触发条件。

## 7. 立案负责人和副组长

### 7.1 任命

- 提案通过后由产品总监任命一名产品经理为立案负责人；
- 立案负责人原则上是未来项目组组长；
- 非产品经理来源提案必须指定一名原提案组成员为副组长；
- 产品经理本人提案不强制设置副组长；
- 原提案组成员继续作为立案协作成员参与，但只获得与其任务和数据等级匹配的权限；
- 任命完成前不能进入正式立案评估任务。

### 7.2 换人

- 产品总监可以更换立案负责人或副组长；
- 更换只要求记录前后人员、操作人和时间；
- 原负责人历史操作和确认不改写；
- 新负责人获得后续处理权限，原负责人权限按项目身份重新计算。

## 8. 立案评估

### 8.1 评估范围

立案阶段由产品部组织补充：

- 生产方；
- 合作模式；
- 工厂信息；
- 基本工艺技术路线；
- 原料、包材和供应可行性；
- 产品成本构成；
- 建议排期和资源占用；
- 主要风险和前置条件。

字段和模板可按品类配置，但上述评估类别必须保留。

### 8.2 任务与责任

- 立案负责人是该阶段唯一 A；
- 部门责任模板生成评估任务；
- 部门负责人落实具体个人 R；
- 一个原子任务只设置一个 R；
- 关键评估文件可以要求部门负责人或指定人员专业确认；
- 所有正式评估文件必须形成受控版本。

### 8.3 提交立项评审

系统必须检查：

- 核心评估类别已有结论；
- 核心必交付物齐全或存在有效豁免；
- 要求专业确认的文件已确认；
- 项目建议排期、负责人建议和资源风险已填写；
- 评审引用的文件版本已锁定。

未通过检查时不得提交，并明确列出阻塞项。

## 9. 立项评审和项目创建

### 9.1 重大阶段门

“立案进入立项”是固定重大阶段门，记录规则与提案阶段一致。

### 9.2 立项结果

| 结果 | 系统动作 |
|---|---|
| 通过 | 创建项目实例并初始化项目工作台 |
| 待补充 | 返回立案评估补充后重新提交 |
| 暂缓推进 | 进入候选机会池并保留当前立案成果 |
| Pass | 锁定立项评审版本，可未来复议 |

### 9.3 通过后的原子操作

立项通过必须在一次业务事务中完成：

1. 创建正式项目实例；
2. 将立案负责人转为项目组组长；
3. 保留副组长和原提案组成员；产品总监可结合实际排期调整项目成员；
4. 复制项目类型模板快照；
5. 初始化阶段、任务、交付物和计划；
6. 新品创建“研发中产品”和草稿初始版本；
7. 老品迭代关联已有产品并创建产品变更记录；
8. 建立机会资产、项目和产品之间的来源关系；
9. 生成项目创建审计记录和通知。

任一步骤失败时不得形成半创建状态；系统应回滚并允许授权人员重试。

## 10. 多提案合并和单提案拆分

### 10.1 合并

- 产品总监可以在立项前提出将多个机会资产合并为一个项目；
- 各机会资产保持独立历史、贡献人和评审记录；
- 合并评审材料必须展示所有来源机会；
- 合并后只创建一个项目实例；
- 提案额度和贡献记录不因合并重算。

### 10.2 拆分

- 产品总监可以将一个机会资产拆分为多个拟立项方案；
- 每个方案单独形成立项评审版本；
- 通过的方案分别创建项目实例；
- 未通过方案保留自身评审结果；
- 所有项目均能回溯到原机会资产。

### 10.3 新品与老品分流

提案阶段的类型仅为初判。立项评审时必须明确每个拟立项方案是：

- 创建新品；
- 迭代已有产品；
- 暂不形成项目。

## 11. 暂缓池和季度回看

暂缓记录至少包含：

- 当前停留阶段；
- 最近评审结论；
- 暂缓原因；
- 重启触发条件；
- 当前责任人；
- 下次季度回看时间。

季度回看结果限定为：

- 继续暂缓；
- 重启评审；
- 转为 Pass；
- 调整触发条件。

重启评审默认返回原停留阶段，不能直接跳过对应阶段门。

## 12. Pass 和复议

- 原 Pass 记录不可编辑；
- 复议创建新的评审版本，并关联原记录；
- 原正式提案负责人或当前具备提案资格的人可以发起；
- 普通联合成员不能独立发起；
- 原负责人离职、转岗或失去资格时，产品总监可指定新负责人；
- 默认回到被 Pass 的同一阶段；
- 产品总监可因前置判断变化退回更早阶段；
- 原提案贡献和额度归属不改写。

## 13. 页面需求

### 13.1 提案入口

- 我的草稿；
- 我负责的提案；
- 我参与的联合提案；
- 新建提案；
- 季度额度提示。

### 13.2 机会工作台

- 当前阶段和状态；
- 提案/评估版本；
- 成员和负责人；
- 任务、交付物和文件；
- 阶段门记录；
- 暂缓、Pass 和复议历史；
- 来源与关联产品/项目。

### 13.3 经管会评审页

- 决策摘要；
- 关键材料和版本；
- 经管会整体结论录入；
- 老板最终决策录入；
- 差异提示；
- 决策后状态预览。

### 13.4 候选机会池

支持按停留阶段、责任人、季度回看状态、触发条件和最近评审时间筛选。

## 14. 权限与可见性

- 未立案提案默认仅提案组、产品总监、经管会授权人员可见；
- 联合成员可查看其参与提案的完整内容和评审结论；
- 普通员工可查看已立案机会的标题和公开摘要；
- 普通员工可继续查看自己提报或参与的 Pass、暂缓记录；
- 联合成员不因进入立案而自动获得成本、供应商、工艺等敏感资料权限；
- 立案和立项全文根据项目身份、角色、部门和敏感等级授权；
- 下载、导出和通知摘要受同一权限策略控制。

## 15. 通知

系统至少发送：

- 联合成员邀请和接受结果；
- 提案提交、退回、待补充、暂缓、Pass 和通过；
- 立案负责人、副组长任命或更换；
- 评估任务和专业确认待办；
- 重大阶段门待处理和决策结果；
- 暂缓池季度回看提醒；
- 复议创建和结果。

钉钉只发送摘要和系统深链接，不在钉钉内直接决策。

### 15.1 异常与边界

- 同一提案重复提交时只允许一次状态转换，其余请求返回当前状态；
- 提案负责人离职或失去资格后，草稿不能提交，由产品总监指定新负责人；
- 重大决策角色未配置时不得形成阶段门结果；
- 合并或拆分期间来源机会发生新决策时，必须重新确认关联方案；
- 已被阶段门引用的提案或评估版本不得删除或覆盖；
- 项目创建成功但通知失败时不回滚业务，系统待办保持有效并记录通知失败；
- 审计记录写入失败时，提交、决策、合并、拆分和复议操作必须失败。

## 17. 验收场景

### 17.1 产品经理独立提案

产品经理完成四项核心内容并提交，经重大阶段门通过后进入立案；产品总监任命立案负责人，完成评估并立项，系统创建项目、研发中产品和草稿初始版本。

### 17.2 普通员工联合提案

普通员工只能受邀加入合格提案组，能协作和查看本人参与提案，但不能独立提交或发起复议；进入立案后不自动看到敏感评估资料。

### 17.3 暂缓与重启

评审结果为暂缓推进，只填写重启触发条件亦可保存；记录进入候选机会池，在季度回看时重启并返回原阶段评审。

### 17.4 Pass 与复议

原 Pass 版本保持不可修改；合格人员发起复议后生成新版本，默认返回原阶段，全部新旧记录可追溯。

### 17.5 合并与拆分

两个提案合并创建一个项目时，各提案历史和贡献不丢失；一个提案拆分创建两个项目时，两个项目均能回溯原机会。

### 17.6 创建失败

立项通过后的对象初始化任一步失败时，不产生半成品项目或产品资产；授权用户可根据明确错误重试。
