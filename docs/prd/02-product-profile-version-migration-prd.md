# 子 PRD 02：产品档案、版本与存量迁移

版本：V0.1  
日期：2026-06-29  
状态：已确认基线

基线确认日期：2026-07-02
上游依据：总体 PRD、产品全生命周期架构设计说明

## 1. 目标与范围

本模块建立全公司统一产品档案，使用户能够快速获取产品当前有效信息，并在开发、迭代和迁移过程中持续完善档案。

首期范围包括：

- 产品资产、产品版本、SKU 和渠道配置；
- 固定核心字段与可配置属性组；
- 营养成分、包装图片和受控文件素材；
- 项目草稿、差异比较、确认和发布；
- 产品版本、SKU 和渠道配置生效历史；
- 存量老品导入基线；
- 产品档案检索、查看和权限控制；
- 外部系统编码和数据来源绑定。

不包括 ERP、MES、WMS 主数据的替代维护。

## 2. 用户与职责

| 角色 | 职责 |
|---|---|
| 普通员工 | 查看公开或内部可见的产品档案 |
| 产品经理/项目组长 | 在项目中维护产品档案草稿 |
| 产品总监 | 确认老品基线、修正录入错误、批准普通迭代生效 |
| 部门负责人/专业确认人 | 确认本部门负责的关键属性组或文件版本 |
| 项目成员 | 按任务编辑获授权属性 |
| 经营监督人 | 查看与经营监控相关的 SKU、渠道和状态 |
| 系统管理员 | 配置属性组、字段字典和数据来源，不默认查看敏感值 |

### 2.1 关键用户故事

- 作为普通员工，我可以快速搜索产品并查看权限范围内的统一档案。
- 作为项目组长，我可以在项目草稿中逐步完善产品信息，而不影响当前有效档案。
- 作为专业确认人，我可以确认具体属性组和文件版本，并在内容变化后重新确认。
- 作为产品总监，我可以发布老品导入基线或批准普通迭代生效。
- 作为系统管理员，我可以配置属性结构，但不能借此读取高敏业务值。

### 2.2 通用前置条件

- 产品、版本、SKU 和渠道字典已初始化；
- 属性组、字段类型、敏感等级和确认规则已发布；
- 文件存储和版本服务可用；
- 项目草稿已关联合法项目或迁移批次；
- 外部编码尚未接入时允许为空，但一旦绑定必须通过唯一性校验。

## 3. 产品档案对象模型

```text
产品资产
  └─ 产品版本
      └─ SKU
          └─ 渠道配置
```

### 3.1 产品资产

代表长期存在的产品身份，不因版本升级或项目结束而消失。

固定核心字段至少包括：

- 内部产品标识；
- 产品名称；
- 品牌；
- 品类；
- 产品负责人；
- 来源类型：新品立项、存量产品导入；
- 生命周期状态；
- 当前有效版本；
- 创建时间和创建来源；
- 外部系统关联标识。

### 3.2 产品版本

代表产品核心定义在某一时期的版本。涉及配方、核心规格、营养、保质期、核心包装定义或其他核心事实变化时创建新版本。

状态包括：

- 草稿；
- 待确认；
- 已批准待生效；
- 有效；
- 已停用。

同一产品可存在多个有效版本，用于渠道分支、库存过渡或并行销售，但必须明确适用范围和生效时间。

### 3.3 SKU

代表实际售卖、库存和效期管理单元。至少记录：

- SKU 名称；
- 所属产品版本；
- 规格和净含量；
- 售卖单位；
- 内外包装形式；
- 装箱关系；
- 条码或拟定条码；
- 状态；
- 生效和停用时间；
- ERP/WMS 等外部编码。

### 3.4 渠道配置

代表 SKU 在具体渠道下的销售设置。至少记录：

- 渠道；
- 建议零售价；
- 渠道卖点；
- 渠道状态；
- 上市/停售时间；
- 渠道素材；
- 监控指标适用范围；
- 外部渠道商品标识。

价格、状态和素材变化形成带生效时间的变更记录，不覆盖历史。

## 4. 属性组模型

### 4.1 设计原则

产品档案采用“固定核心字段 + 可配置属性组”：

- 核心对象和关键字段固定，保证统一检索和统计；
- 不同品类可增加属性组或扩展字段；
- 属性组不能改变产品—版本—SKU—渠道层级；
- 不使用完全自由的万能字段替代核心数据结构。

### 4.2 首期基础属性组

| 属性组 | 归属层级 | 典型内容 |
|---|---|---|
| 基本信息 | 产品 | 名称、品牌、品类、定位、负责人、状态 |
| 产品定义 | 产品版本 | 目标人群、核心卖点、消费场景、产品描述 |
| 配方与原料 | 产品版本 | 原料、配方摘要、过敏原、关键限制 |
| 营养成分 | 产品版本 | 营养素、每 100g/100ml 含量、单位、NRV% |
| 质量与合规 | 产品版本 | 执行标准、保质期、储存条件、合规说明 |
| 包装规格 | SKU | 净含量、包材、箱规、包装尺寸和条码 |
| 渠道经营 | 渠道配置 | 建议零售价、渠道卖点、渠道状态和素材 |

属性组模板可配置适用品类、字段、数据类型、是否必填、敏感等级、负责部门和是否需要专业确认。

### 4.3 字段类型

首期至少支持：

- 单行/多行文本；
- 整数和小数；
- 金额和百分比；
- 日期；
- 单选和多选字典；
- 布尔值；
- 人员和部门；
- 产品、版本、SKU、渠道等对象引用；
- 图片和文件引用；
- 结构化明细表。

字段配置变更形成版本，只影响新建草稿或明确执行模板升级的对象，不静默改变历史有效档案。

## 5. 营养成分和包装素材

### 5.1 营养成分

营养成分必须同时支持：

- 结构化明细：营养素名称、含量、单位、NRV% 和检测/计算来源；
- 对应标签图片或正式文件；
- 适用产品版本；
- 确认状态和确认版本。

同一营养成分表不得只保存为不可检索图片。结构化值和标签图片不一致时阻止发布，并提示专业确认人处理。

### 5.2 包装素材

包装素材至少区分：

- 内包装图片；
- 外包装图片；
- 标签稿；
- 设计源稿；
- 渠道展示图；
- 已批准印刷稿。

每项素材关联产品版本或 SKU，记录文件版本、用途、敏感等级、有效状态和确认人。已批准素材修订后生成新版本，不覆盖旧文件。

## 6. 项目草稿

### 6.1 草稿创建

- 新品立项通过后创建产品资产和草稿初始版本；
- 老品迭代立项通过后创建产品变更记录，并按变更范围创建草稿版本、SKU 或渠道配置；
- 项目组只编辑与本项目关联的草稿，不直接修改当前有效档案；
- 草稿预填当前有效值并标识本次拟变更字段。

### 6.2 编辑权限

- 项目组长拥有草稿整体维护和提交权限；
- 任务 R 可编辑分配给本人的属性组；
- 专业确认人可以查看负责属性组和关联依据；
- 敏感字段继续受数据等级和专项授权限制；
- 所有编辑记录操作人、时间和前后值。

### 6.3 差异比较

系统必须提供草稿与当前有效档案的差异视图，至少显示：

- 新增、修改和删除字段；
- 旧值和新值；
- 所属属性组；
- 编辑人和最后更新时间；
- 专业确认状态；
- 关联交付物或文件版本。

## 7. 确认、批准和发布

### 7.1 属性组确认

- 只有模板配置为关键的属性组要求专业确认；
- 确认动作是“确认通过”或“退回修改”；
- 确认绑定具体草稿版本；
- 草稿中已确认字段再次变化时，该属性组自动恢复待确认；
- 产品总监或项目组长可调整确认人并留痕。

### 7.2 发布条件

发布前系统必须检查：

- 必填核心字段完整；
- 关键属性组已经确认；
- 引用的正式文件为锁定版本；
- 生效时间和适用范围明确；
- 对象之间不存在无效引用；
- 对应阶段门或产品总监批准已完成。

### 7.3 发布权限

- 新品首次上市由重大阶段门批准后发布；
- 老品普通迭代由产品总监批准后发布；
- 退市由重大阶段门批准后更新产品、版本、SKU 和渠道状态；
- 系统管理员无权代替业务批准发布。

### 7.4 发布结果

发布必须原子化完成：

1. 锁定本次草稿；
2. 创建或更新有效版本、SKU、渠道配置；
3. 写入生效和适用范围；
4. 保留旧版本；
5. 更新产品档案当前视图；
6. 记录批准依据、文件版本和审计日志。

失败时不得出现部分字段已生效、部分字段未生效的状态。

## 8. 产品档案展示

### 8.1 概览

优先展示用户最常使用的信息：

- 产品名称、图片、品牌、品类和状态；
- 当前有效版本；
- 规格/SKU；
- 建议零售价和渠道状态；
- 保质期和储存条件；
- 核心卖点和营养摘要；
- 产品负责人；
- 最近一次迭代和更新时间。

### 8.2 详情页

详情页按权限展示：

- 基本信息；
- 产品定义；
- 配方与营养；
- 质量与合规；
- SKU 和包装；
- 渠道配置；
- 图片和文件；
- 经营监控；
- 历史版本、项目和决策来源。

### 8.3 搜索和筛选

至少支持按名称、品牌、品类、状态、负责人、SKU、外部编码和渠道搜索或筛选。用户只能搜索到有权查看的产品和字段。

## 9. 存量产品导入

### 9.1 导入方式

- 单个表单录入；
- Excel 批量导入；
- 后续外部系统接口同步。

首期必须提供可下载的标准导入模板和导入结果报告。

### 9.2 导入流程

```text
上传/录入 → 字段校验 → 重复识别 → 预览 → 确认导入
→ 建立产品资产和导入基线版本 → 补充完善 → 产品总监发布
```

### 9.3 重复识别

系统按以下信息提示潜在重复：

- 外部产品或物料编码；
- 产品名称和规格；
- 条码；
- 品牌、品类和净含量组合。

重复提示不自动合并。授权人员选择关联已有产品、跳过或继续新建，并留下处理记录。

### 9.4 数据完整状态

导入基线状态为：

- 完整；
- 部分完整；
- 待补充。

资料不全不阻止收录。系统应显示缺失属性组和完成度提示，但不建立复杂评分审批。

### 9.5 基线发布

- 产品总监一次确认后发布当前有效基线；
- 经管会不参与首次收录；
- 历史文件标记为“迁移历史资料”，不补造阶段门；
- 补充缺失图片、历史文件和来源说明只留痕；
- 修正明显录入错误由产品总监确认；
- 改变有效产品事实必须创建产品迭代项目。

## 10. 数据主权与外部绑定

| 数据 | 本系统行为 |
|---|---|
| 产品定义、版本、营养、包装资料 | 本系统为权威来源 |
| ERP 物料/SKU 编码、BOM 执行数据 | 保存引用和同步快照 |
| MES 生产执行和批次工艺参数 | 只读引用 |
| WMS 库存、批次和实际到期时间 | 只读引用 |
| 销售/BI 原始交易和经营数据 | 只读引用并用于监控 |

外部字段至少展示来源系统、外部标识和最近同步时间。数据冲突按字段主权处理，不允许普通用户直接覆盖外部权威事实。

## 11. 权限和敏感信息

- 公开摘要可向普通员工展示；
- 内部产品基本信息按公司内部权限展示；
- 配方、成本、供应商、工艺和未上市包装属于受控或高敏信息；
- 项目草稿仅授权项目成员和审批/确认角色可见；
- 图片预览、原文件下载和批量导出分别授权；
- 系统管理员只能配置属性结构，默认看不到敏感字段值；
- 历史版本继承当时的数据等级，不因当前版本公开而自动公开。

## 12. 异常与边界

- 当前有效版本被其他项目同时变更时，后提交项目必须看到基线已变化提示并重新比较；
- 外部编码绑定冲突时阻止发布；
- 删除被 SKU、渠道或决策引用的版本时，只允许停用；
- 文件上传失败不改变草稿字段和确认状态；
- 发布失败保持草稿和批准记录，授权用户可修复后重试；
- 产品合并不在首期范围，重复产品通过人工确认后分别保留或停止维护。

## 14. 验收场景

### 14.1 新品档案发布

立项通过后自动创建研发中产品和草稿初始版本。项目成员按任务完善属性组，关键内容完成专业确认；首次上市批准后一次性发布有效产品档案。

### 14.2 老品基线导入

通过 Excel 导入部分完整的老品、SKU 和包装图片，系统提示潜在重复并生成结果报告；产品总监确认后发布导入基线，不要求补造历史项目。

### 14.3 普通迭代

迭代项目从当前有效档案生成草稿，只修改包装和渠道配置；产品总监批准后新配置生效，旧记录和原包装文件仍可追溯。

### 14.4 敏感字段

普通员工可以看到产品基本信息和公开图片，但不能看到配方、成本和高敏文件；系统管理员可以配置“配方”字段但不能读取其业务值。

### 14.5 发布失败

发布过程中任一对象写入失败时，有效产品档案保持原状，草稿和批准依据保留，可在修复后重新发布。
