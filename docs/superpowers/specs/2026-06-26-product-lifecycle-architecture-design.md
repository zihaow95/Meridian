# 产品/项目全生命周期管理系统架构设计说明

版本日期：2026-06-30

适用范围：本设计文档用于产品总监、经管会及系统建设团队确认产品/项目全生命周期管理系统的总体架构、核心对象、流程主干、权限边界和首期建设范围。

## 1. 系统定位

本系统定位为“产品全生命周期治理平台”，不是单纯的提案审批系统、普通项目管理系统或文档管理系统。

系统主干固定为：

```text
提案 → 立案 → 立项 → 开发 → 上市 → 运营/退市
```

系统从产品维度维护公司内部产品从机会产生到生命周期末期的全过程，形成可追溯、可监管、可复盘、可扩展的产品数字主线。

ERP、MES、WMS、销售系统、财务系统等已有系统继续负责各自专业数据。本系统不取代这些系统，而是负责把产品生命周期中的流程、决策、文件、责任、数据引用和经营监控串起来，为老板、经管会、产品总监、项目组和相关业务部门提供统一工作入口。

## 2. 总体架构原则

### 2.1 主干完整，首期分深浅实现

首期不能只做“提案—立案—立项”，否则系统会退化为提案审批工具。首期应保证完整生命周期真实可运行：

- 提案、立案、立项：深度实现业务规则。
- 开发、上市、运营/退市：流程完整、交付物完整、文件归档完整、阶段门完整，但专业模板先通用化。

### 2.2 产品资产是主线，项目是治理事件

产品资产长期存在，项目是推动产品变化的一次治理事件。

- 新品开发项目在立项通过后创建“研发中产品”及草稿初始版本。
- 产品迭代项目修改、升级或派生已有产品资产。
- 项目结束后不消失，而是沉淀为产品资产履历的一部分。

### 2.3 前台使用“产品迭代项目”，后台按“产品变更”建模

用户侧使用“产品迭代项目”这一更中性的表达。系统底层按产品变更能力建模，以覆盖升级、渠道定制、包装变更、降本、调价、供应调整、暂停销售、退市等场景。

### 2.4 统一主干 + 场景化策略

新品开发、老品迭代、渠道定制、降本、包装变更、紧急变更等都必须挂在统一主干上。不同场景只能调整交付物、任务、确认强度、复用策略、简化策略和豁免规则，不能脱离主干形成独立流程。

### 2.5 文档承载完整过程，结构化记录承载关键决策

项目过程以文档和文件为主。阶段门、决策依据、责任人、结论、例外、审计必须结构化记录。

### 2.6 权限采用 RBAC + ABAC + 操作留痕

权限不是单纯控制“能不能看”，还要控制字段、页面区块、下载、导出、分享、授权和关键操作行为。

## 3. 核心对象模型

### 3.1 产品机会资产

立项前的提案、立案、立项候选不是正式产品资产，但仍是公司的重要知识资产。系统底层以“产品机会资产”统一串联这些记录。

产品机会资产覆盖：

- 提案；
- 立案评估；
- 立项候选；
- 待补充记录；
- 暂缓推进记录；
- Pass 记录；
- 复议版本；
- 相关文件、评估和决策依据。

产品机会资产编号只作为系统内部关联 ID，不向终端用户暴露。用户前台仍然看到提案、立案、立项候选、暂缓池、Pass 记录和复议记录。

产品机会资产与项目实例采用多对多关系：

```text
一个或多个产品机会资产 → 一个项目实例
一个产品机会资产 → 一个或多个项目实例
```

这支持多个提案合并立项、一个提案拆分为多个项目，以及同一提案最终分别形成新品或老品迭代。原产品机会资产不合并、不覆盖，继续作为项目和产品资产的来源履历保留。

### 3.2 产品资产

新品立项通过后创建正式产品资产，初始状态为“研发中产品”，同时创建草稿初始版本。老品迭代立项通过后关联已有产品资产并创建产品变更记录，不重复创建产品资产。

产品资产层级为：

```text
产品资产
  └─ 产品版本
      └─ SKU
          └─ 渠道销售单元 / 渠道配置
```

含义：

- 产品资产：长期存在的产品主对象。
- 产品版本：新品初始版本或老品迭代后的版本。
- SKU：实际售卖、库存、效期管理单元。
- 渠道销售单元/渠道配置：同一 SKU 在不同渠道下的销售配置、价格、状态、监控指标。

变更边界：

- 影响产品核心定义的变化，创建草稿产品版本。
- 影响实际售卖、库存或效期管理单元的变化，创建或调整草稿 SKU。
- 影响渠道策略、渠道价格、渠道状态、展示素材等，形成带生效时间的渠道配置变更。
- 只影响项目执行过程的变化，作为管理调整，不改变产品事实。

草稿产品版本、SKU 或渠道配置通过对应的上市/变更生效阶段门后才成为有效事实。历史版本不覆盖，可以按产品实际情况停用或并行保留。

### 3.3 项目实例

项目实例在立项通过后创建。立项前由产品机会资产承载提案、立案和立项候选过程。

前台项目类型：

- 产品项目：主要面向新品开发。
- 产品迭代项目：主要面向已有产品的升级、优化、渠道定制、降本、包装变更、调价、退市等。

后台可以统一抽象为产品项目和产品变更项目。

项目实例承载：

- 当前阶段；
- 阶段处理方式；
- 项目组；
- 任务；
- 交付物；
- 文件版本；
- 专业确认；
- 阶段门；
- 例外通过；
- 审计记录。

生命周期看板通过统一视图同时展示立项前的产品机会资产和立项后的项目实例。用户看到的是连续主干，底层不强行把提案伪装成正式项目。

### 3.4 统一产品档案

产品档案是绝大多数用户获取产品信息的主要前端载体，也是产品资产的核心呈现。产品开发和产品迭代本质上是在持续完善、确认并发布产品档案。

产品档案采用“固定核心字段 + 可配置属性组”的混合模型：

- 产品级：名称、品牌、品类、负责人、状态、来源等；
- 产品版本级：配方、原料、营养成分、保质期、储存条件、核心卖点、执行标准等；
- SKU 级：规格、净含量、条码、内外包装形式、装箱关系等；
- 渠道级：建议零售价、渠道卖点、渠道状态、渠道素材等；
- 文件素材：内外包装图片、标签稿、规格书、检测报告等，并关联具体产品版本或 SKU。

营养成分同时保存结构化数据和对应标签图片/正式文件。属性组可以按品类配置字段、负责部门、确认人和敏感等级，但不能改变产品—版本—SKU—渠道的统一主模型。

项目中的产品信息先进入项目草稿：

```text
项目草稿 → 属性组确认 → 阶段门/变更生效 → 发布到有效产品档案
```

开发中或变更中的信息未经产品总监或对应重大阶段门批准，不得更新当前有效产品档案。

### 3.5 存量产品基线迁移

存量老品不补造历史提案、阶段门和项目记录，直接创建来源为“存量产品导入”的产品资产及“导入基线版本”。

- 授权人员可通过表单或 Excel 补充产品、版本、SKU、渠道和文件素材；
- 基线可标记为完整、部分完整或待补充，不因资料不全阻止收录；
- 产品总监一次确认后发布当前有效基线，经管会不参与首次档案收录；
- 补充缺失图片、历史文件和来源说明等非事实变更，只需留痕；
- 明显录入错误由产品总监确认修正；
- 基线发布后，配方、营养、规格、包装、保质期、价格等有效产品事实的变化必须进入产品迭代项目。

核心原则是：首次收录用于建立事实基线，后续改变事实必须走迭代流程。

## 4. 生命周期流程设计

主流程固定为：

```mermaid
flowchart LR
  A["提案"] --> B["立案"]
  B --> C["立项"]
  C --> D["开发"]
  D --> E["上市"]
  E --> F["运营/退市"]
```

产品机会资产保留提案、立案和立项候选记录；项目实例保留立项通过后的开发、上市、运营/退市记录，并通过来源关系形成完整生命周期视图。每个项目阶段可以标记为：

- 执行；
- 复用；
- 简化；
- 豁免；
- 不适用；
- 并行处理。

授权规则：

| 阶段处理方式 | 规则 |
|---|---|
| 执行 | 按模板正常推进 |
| 复用 | 项目负责人提出，产品总监确认，并绑定历史依据 |
| 简化 | 项目负责人提出，产品总监确认 |
| 豁免 | 项目负责人提出，产品总监判断；可直接豁免，也可上升经管会 |
| 不适用 | 由项目类型策略预设，项目内不随意修改 |
| 并行处理 | 项目负责人提出，产品总监确认 |

## 5. 阶段门与状态模型

阶段门结果统一为：

- 通过；
- 带例外通过；
- 待补充；
- 暂缓推进；
- Pass。

状态含义：

| 状态 | 含义 |
|---|---|
| 通过 | 满足当前阶段门要求，进入下一阶段或创建下一类对象 |
| 带例外通过 | 存在未满足项，但由产品总监确认后允许进入下一阶段 |
| 待补充 | 材料、方案、评估不完整，需要补充后再评 |
| 暂缓推进 | 方案有价值，但因资源、技术、时机、预算、战略节奏等原因暂不推进 |
| Pass | 当前明确不继续推进，但允许未来复议 |

重大阶段门只记录：

- 经管会整体结论；
- 老板最终决策；
- 决策时间；
- 决策依据摘要；
- 绑定文件版本和数据快照。

不记录经管会成员个人投票或个人意见。

固定重大阶段门限定为四个：

1. 提案进入立案；
2. 立案进入立项；
3. 新品首次上市；
4. 产品退市。

四个重大阶段门由经管会形成整体结论，老板作出最终决策。流程状态以老板最终决策为准；二者不一致时，同时保留并明确标示差异。经管会整体结论和老板最终决策均由具备对应业务权限的指定人员录入或确认。

老品普通迭代的变更生效默认由产品总监确认；需要升级讨论时由产品总监提交经管会，但不新增复杂的风险分级审批。其他非重大阶段门授权具体业务部门负责人或产品总监处理。

“带例外通过”仅限产品总监操作，只记录例外通过说明、确认人、确认时间、阶段门、当时交付物状态和文件版本。非重大阶段门可由产品总监直接确认；重大阶段门中的例外说明不能替代经管会结论和老板最终决策。

## 6. Pass、复议与版本记录

Pass 后允许基于原记录发起重新提交或复议。

规则：

- 原 Pass 记录不可修改，作为历史决策保留。
- 复议生成新版本记录。
- 新版本继承原始记录的基础信息，但需要形成新的评审版本。
- 新旧版本之间建立关联。
- 默认回到被 Pass 的同一阶段重新评审。
- 如产品总监判断复议内容已经影响前置判断，可以退回更早阶段。
- 原提案贡献记录不改写。

复议发起权：

- 原正式提案主负责人可以发起。
- 当前具备提案资格的人可以发起。
- 普通联合提案成员不能绕过提案资格发起复议。
- 原正式提案主负责人离职、转岗或不再具备提案资格时，产品总监可以指定新的具备资格负责人发起复议。

## 7. 候选机会池 / 暂缓池

“暂缓推进”进入候选机会池/暂缓池。它不是失败池，而是公司机会储备池。

适用场景：

- 方案有价值，但产品部排期无法承接；
- 市面工艺或技术条件暂不成熟；
- 产品季节性强，需要控制上市窗口；
- 当前预算、资源或战略节奏不适合推进。

暂缓池记录：

- 暂缓原因；
- 重启触发条件；
- 责任人；
- 当前停留阶段；
- 最近一次评审结论；
- 季度回看状态。

暂缓池默认进入季度回看。季度回看结果限定为：

- 继续暂缓；
- 重启评审；
- 转为 Pass；
- 调整触发条件。

## 8. 提案机制

正式提案至少包含：

- 市场分析；
- 核心卖点；
- 消费人群/用户需求；
- 建议零售价。

提案资格：

- 产品经理可独立提案。
- 销售端口、市场营销等部门由部门负责人组织正式提案。
- 普通员工不能独立提交正式提案，只能作为联合提案成员参与。
- 正式提案主负责人必须是具备提案资格的人。

季度提案指标：

- 产品经理个人提案计入产品经理额度。
- 部门提案计入对应部门额度。
- 联合成员只记录贡献，不重复计数。

联合提案成员：

- 可以查看自己参与提案的 Pass、暂缓、待补充等状态。
- 可以查看完整提案内容和评审结论。
- 不自动获得立案/立项后的敏感资料访问权。
- 后续如被纳入项目组，再根据项目角色获得对应权限。

## 9. 立案、立项与项目组

提案通过后进入立案。

立案开始时：

- 任命产品经理为立案负责人。
- 非产品经理来源的提案，需要从原提案组中指定一名副组长参与。
- 产品部介入评估生产方、合作模式、工厂信息、工艺技术路线、产品成本构成等。

立项通过后：

- 立案负责人默认转为项目组组长。
- 可换人，但必须留痕。
- 创建“研发中产品”。
- 原产品机会资产成为产品资产来源履历。

项目组组长职责：

- 统一推进项目；
- 统一归口交付物；
- 组织任务、文件、专业确认和阶段门提交；
- 对交付物完整性、及时性和归档版本负责。

### 9.1 部门责任模板与个人责任实例

责任模型采用两层设计。

第一层是部门责任模板，用于定义每个阶段事项的：

- 主责部门；
- 执行部门；
- 协同部门；
- 知会部门。

既有部门级 RACI 矩阵只作为模板来源，不直接照搬。原 A、R、A/R、C/R 分别转换为主责、执行、主责兼执行、协同执行，并结合新的提案—立案—立项主干重新整理。

第二层是项目个人责任实例：

- 提案阶段由提案负责人作为唯一 A；
- 立案阶段由立案负责人作为唯一 A；
- 立项通过后由项目组组长作为唯一 A；
- 每个原子任务只设置一名个人 R，需要多个部门执行时拆分为部门子任务；
- C、I 默认保留到部门级，关键事项再指定个人；
- 部门负责人负责从本部门指定或更换 R；
- 项目副组长是协同角色，不形成第二个 A；
- 专业确认人和阶段门处理人是独立职责，不混入 RACI。

部门参与关系用于模板生成、资源协调和默认通知，不自动向部门全员授予敏感数据权限。个人被正式分派后，系统再按职责和数据敏感等级授予必要权限。

## 10. 任务与交付物模型

任务和交付物分开设计。

任务用于协作推进：

- 谁在什么时候做什么；
- 可有负责人、截止时间、状态；
- 不一定产生正式交付物。

交付物用于阶段门证据：

- 阶段门检查交付物和审批状态；
- 正式交付物必须上传受控版本归档；
- 阶段门引用具体版本，不引用“当前文件”。

交付物三层模型：

| 类型 | 规则 |
|---|---|
| 核心必交付物 | 公司强制要求，不能随意删除，只能申请豁免 |
| 模板推荐交付物 | 项目类型带出，可调整 |
| 项目自定义交付物 | 项目组根据实际新增 |

交付物责任：

- 交付物归口负责人是项目组组长。
- 具体编制人可以是任务执行人或业务部门成员。
- 关键专业交付物可配置专业确认。

模板发布后形成不可变模板版本。创建项目时复制对应模板快照，后续模板调整只影响新创建项目，不反向修改已有项目。项目组长和产品总监可在快照基础上增加任务、推荐交付物和计划时间，但核心必交付物只能按豁免规则处理。

## 11. 专业确认

专业确认可以理解为业务部门负责人在系统里“签个字”。它不是阶段门决策，也不是经管会审批，而是对交付物中属于本部门专业职责范围的内容做确认。

规则：

- 只对模板配置为需要专业确认的关键交付物启用。
- 默认由对应业务部门负责人本人确认。
- 产品总监可指定专业确认人。
- 产品总监和项目组长可调整专业确认人。
- 指定和调整只做操作留痕，不强制填写原因。
- 专业确认绑定具体交付物版本。
- 交付物生成新受控版本后，如阶段门引用新版本，必须重新专业确认。

确认动作轻量化：

- 确认通过；
- 退回修改。

## 12. 任务调整与逾期处理

任务调整规则：

| 调整类型 | 规则 |
|---|---|
| 项目组内部轻微任务调整 | 项目组长直接调整并留痕 |
| 跨部门任务调整 | 产品总监确认 |
| 影响关键交付物、专业确认、阶段门、关键里程碑 | 产品总监确认 |
| 产品总监无法协调的资源冲突 | 上升经管会 |

轻微任务调整定义：

- 不改变交付物范围；
- 不影响阶段门；
- 不涉及跨部门资源；
- 不影响关键节点；
- 不改变专业确认人。

首期逾期处理只做提醒和看板标红：

- 任务逾期：提醒负责人和项目组长，项目工作台标红。
- 交付物逾期：提醒项目组长，阶段看板标红。
- 专业确认逾期：提醒确认人和项目组长，交付物状态标红。
- 阶段门逾期：提醒项目组长和产品总监，项目总览标红。

首期不自动升级经管会，不自动冻结流程，不自动改变项目状态。

紧急情况下允许总监级人员发起“先执行、后补确认”：

- 仅限总监级人员操作；
- 必须绑定具体项目、事项和待补确认内容；
- 系统持续提醒并在看板标红；
- 超期未补确认时升级提示经管会；
- 首期不自动冻结业务或回滚现实操作。

## 13. 运营监控与产品迭代触发

运营监控链路为：

```text
销售/经营数据 → 风险信号 → 经营议题 → 研判结论 → 产品迭代项目
```

系统只自动生成风险信号，不自动启动迭代项目。

风险信号来源：

- 销售下滑；
- 用户净流出；
- 竞品抢占；
- 连续一个季度无法消化“四分之一效期最低生产量”；
- 其他经营异常。

经营议题轻量化：

- 默认责任人是经管会经营监督人。
- 经营监督人可以直接关闭轻微议题。
- 不做复杂风险等级审批。
- 关闭只记录理由和审计。
- 经营议题可转产品迭代项目，也可继续观察、调价、渠道调整、暂停销售、退市等。

经营监督人权限边界：

- 默认拥有经营监控和经营议题权限。
- 不自动拥有成本、供应商、工艺、立项全文等敏感资料权限。
- 其他访问权来自经管会成员、总监、专项授权等角色。

## 14. 经营数据接入

经营监控数据接入同时支持：

- 自动接口同步；
- Excel/CSV 定期导入；
- 手动维护/修正。

统一链路：

```text
数据源 → 接入批次 → 暂存校验 → 标准化映射 → 经营监控事实数据 → 风险信号
```

数据粒度：

```text
SKU × 渠道 × 时间
```

同一数据出现接口值和人工值时：

- 允许人工确认后的数据覆盖接口数据。
- 不覆盖原始数据，而是生成“人工确认有效值”。
- 有效值持续有效，直到再次修改或撤销。
- 风险信号和看板使用当前有效值。
- 页面标记该数据存在人工确认。
- 经营监督人可在其权限范围内直接修改数据。

人工确认有效值仅用于本系统的经营判断和展示，不改变外部专业系统作为原始经营事实权威来源的地位。

### 14.1 数据主权

数据主权采用“产品定义与治理归本系统，经营和执行事实归专业系统”的原则：

| 数据范围 | 权威来源 |
|---|---|
| 产品档案、版本、规格定义、营养成分、包装资料、生命周期状态、项目决策和受控文件 | 本系统 |
| 正式物料/SKU 编码、BOM 执行数据、采购与财务成本、订单和结算 | ERP |
| 生产执行、批次工艺参数和实际产出 | MES |
| 库存、批次、实际到期时间和出入库 | WMS |
| 销售、渠道、用户和经营分析原始数据 | 销售/BI 系统 |

本系统保存外部数据的来源标识、同步时间和决策所需快照，不把引用数据变成新的权威源。项目阶段可以维护拟定编码、成本和工艺草稿，专业系统正式建档后再绑定实际数据。数据冲突按字段主权确定有效值，不在系统间直接互相覆盖。

## 15. 权限体系

权限采用：

```text
RBAC + ABAC + 操作留痕
```

RBAC 负责角色：

- 系统管理员；
- 产品总监；
- 产品经理；
- 项目组长；
- 项目副组长；
- 项目成员；
- 提案负责人；
- 部门负责人；
- 专业确认人；
- 经管会成员；
- 老板/最终决策人；
- 经营监督人；
- 普通员工。

ABAC 结合：

- 用户角色；
- 部门、岗位、职级；
- 项目身份；
- 产品归属；
- 阶段状态；
- 数据敏感等级；
- 是否专项授权；
- 是否联合提案成员；
- 是否当前责任人。

数据敏感等级：

- 公开摘要；
- 内部可见；
- 项目受控；
- 敏感受控；
- 高敏受控。

权限不仅控制查看，也控制：

- 字段；
- 页面区块；
- 文件正文；
- 下载；
- 导出；
- 分享；
- 通知摘要；
- 授权他人。

普通员工可见边界：

- 可以看到已立案项目的公开摘要。
- 可以看到自己提报或参与的、处于 Pass 或暂缓状态的提案信息。
- 联合提案成员在提案协作期间可以查看其参与提案的完整内容和评审结论。
- 不能看到其他人未立案、Pass、暂缓、待补充的提案。
- 不能看到立案/立项全文、成本、供应商、工艺等受控内容，除非另有项目角色或专项授权。

系统管理员和业务数据访问权必须分开：

- 平台管理员管理账号、配置、集成、运行状态。
- 平台管理员默认不能查看敏感业务内容。
- 排障访问需限时、范围限定、留痕。

关键业务角色和权限由系统管理员人工配置，不因钉钉职级、部门或岗位自动授权。超级管理员是否需要双人复核由系统开关控制；当前指定超级管理员可关闭双人复核，后续更换人员时可以启用。

## 16. 文档与文件治理

项目过程以文档和文件为主。正式交付物必须在系统上传受控版本归档。

文件版本规则：

- 上传形成草稿版本。
- 提交评审后锁定。
- 阶段门引用具体版本。
- 通过后成为正式受控版本。
- 后续修订生成新版本，不覆盖历史。
- 作废只改变有效性，不物理删除历史。

关键决策必须绑定：

- 文件版本；
- 数据快照；
- 决策记录；
- 决策人或确认人；
- 决策时间。

后续文件更新不能改变历史决策依据。

- 消息通知；
- 深链接跳转。

## 19. 技术架构基线

当前规模假设：

- 员工约 200 人；
- 并行项目约 10 个；
- 受控文件约 100GB/年；
- 前期允许 RPO ≤ 24 小时、RTO ≤ 24 小时。

建议采用模块化单体架构，不建议一开始做微服务。

首期按单一企业组织运行，不建设租户开通、计费和跨租户运营能力。核心业务数据保留组织边界字段，文件存储和权限查询按组织范围隔离，为未来行业化应用保留迁移空间。

正式技术基线已在 `2026-06-30-technical-architecture-design.md` 中确认。核心技术结构为：

- Vue 3 + TypeScript 前端 Web 应用；
- Python 3.13 + Django 5.2 LTS + Django REST Framework 模块化单体；
- 独立服务器上的 MySQL 8.0；
- 群晖 DS1621+ 提供的 NFS 受控文件存储；
- Redis + Celery 定时任务和后台任务；
- 钉钉集成适配器；
- 外部数据接入适配器；
- 审计日志模块；
- 权限策略模块。

正式系统不继续沿用旧 MVP 的 Node.js + SQLite 原型技术栈，不同时维护两套后端或两套数据模型。

核心后端模块：

- 身份与权限模块；
- 产品机会资产模块；
- 产品资产模块；
- 产品档案与属性组模块；
- 存量产品基线迁移模块；
- 项目生命周期模块；
- 阶段门模块；
- 任务与交付物模块；
- 文件版本模块；
- 专业确认模块；
- 经营监控模块；
- 经营议题模块；
- 暂缓池/机会池模块；
- 通知模块；
- 审计模块；
- 模板配置模块；
- 数据接入模块。

备份策略：

- 数据库每日备份；
- 文件存储每日备份；
- 备份存放在不同于生产的位置；
- 定期恢复验证；
- 后续可升级为小时级增量备份。

## 20. 首期实现深度定义

首期深度实现“提案—立案—立项”。

深度不是页面多，而是业务闭环真实可用。

首期必须深度实现：

- 提案资格；
- 联合提案；
- 提案额度；
- 部门提案；
- 提案可见性；
- 经管会整体结论；
- 老板最终决策；
- 待补充、暂缓、Pass、复议；
- 立案负责人；
- 产品部评估；
- 立项通过后创建研发中产品；
- 产品机会资产与项目实例多对多关联；
- 新品草稿初始版本和老品产品变更记录；
- 项目组生成；
- 项目工作台初始化；
- 交付物模板带出；
- 文件受控归档；
- 阶段门记录；
- 权限和审计。

统一底座还必须实现：

- 组织、用户、角色和项目身份；
- 部门责任模板和个人责任实例；
- 模板版本与项目快照；
- 产品档案、可配置属性组和草稿发布；
- 产品、版本、SKU、渠道配置及生效历史；
- 存量产品导入基线；
- 文件/素材版本和数据来源标识。

开发、上市、运营/退市阶段首期做到：

- 可以进入、推进、退回、完成、挂起；
- 有阶段门；
- 有任务和交付物；
- 有文件版本；
- 有专业确认能力；
- 有逾期提醒和标红；
- 有经营监控基础链路；
- 有风险信号和经营议题；
- 有暂缓池季度回看。

这些阶段的专业模板可以先通用化，后续逐步增强。

首期不建设复杂流程设计器、ERP/MES/WMS 深度双向集成、自动升级处罚、复杂风险分级、在线文档协作和微服务拆分。

首期至少通过两个端到端业务场景验收：

1. 一个新品从提案完整推进到上市运营；
2. 一个老品从经营风险信号推进到迭代生效或退市。

两个场景都必须验证权限隔离、部门和个人责任、文件版本、产品档案发布、决策依据和完整审计记录。

## 21. 当前架构解决的问题

本架构主要解决以下问题：

- 避免系统退化成提案审批工具；
- 避免产品资产和项目过程断裂；
- 避免老品迭代被迫走新品全流程或完全脱离流程；
- 避免文档归档和阶段决策脱节；
- 避免权限只按角色粗放控制；
- 避免钉钉和系统双状态源；
- 避免后续开发、上市、运营阶段补功能时长出新子系统；
- 避免暂缓、Pass、复议成为不可追溯的口头管理；
- 避免经营监控自动误触发产品迭代。

本设计已经覆盖并确认总体架构、核心对象、状态边界、责任模型、权限原则和首期范围。以下事项不阻塞核心架构和底层开发，但应在对应功能验收前形成默认配置：

- 每个阶段的最小核心交付物清单；

- 首期后台可维护字段和模板范围；
