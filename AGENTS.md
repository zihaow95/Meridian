# Project Meridian Agent Instructions

这些规则适用于本仓库所有任务，除非用户明确覆盖。

## 1. 编码前先思考

- 明确假设、目标和成功标准；
- 有实质歧义时先说明，不猜测业务规则；
- 存在多种解释时列出差异；
- 发现更简单且满足目标的方法时应主动指出；
- 不理解现有结构时停止并调查。

## 2. 简单优先

- 只实现当前已批准需求；
- 不为单次使用创建抽象；
- 不提前建设微服务、BPM、Kubernetes、Harbor或搜索集群；
- 如果实现明显复杂于需求，应先简化。

## 3. 精确修改

- 只修改任务必需文件；
- 不顺手重构无关代码；
- 不覆盖用户已有变更；
- 遵守当前代码风格和模块边界；
- 清理自己引入的临时文件。

## 4. 目标驱动

- 每个任务先定义可验证的完成条件；
- 持续迭代直到验证通过；
- 未执行的检查必须明确披露；
- 不以“代码已写”代替“目标已实现”。

## 5. 确定性工作交给代码

- 模型用于判断、设计、归纳和审查；
- 路由、重试、转换、校验和可重复检查使用代码；
- 能通过脚本验证的内容不得仅靠人工判断。

## 6. 控制上下文和任务范围

- 大任务按独立切片拆分；
- 每个切片保持可运行、可测试；
- 接近上下文限制时先形成检查点；
- 不静默缩小用户目标。

## 7. 冲突必须显式处理

- 两种模式冲突时选择较新、已确认和有测试的一种；
- 说明选择原因；
- 标记旧模式待归档；
- 不混合Node/SQLite旧原型与Django/MySQL正式架构。

## 8. 写前先读

修改模块前至少阅读：

- 对应PRD和TRD；
- 模块公开接口；
- 直接调用方；
- 共享工具和测试；
- 当前迁移和配置版本。

## 9. 测试表达业务意图

- 先写失败测试，再写最小实现；
- 测试名称说明规则为什么存在；
- 权限同时测试允许和拒绝；
- 状态同时测试合法和非法迁移；
- 并发测试检查最终数据库事实；
- 不以SQLite替代MySQL关键行为。

## 10. 重要步骤后形成检查点

每完成一个显著步骤，记录：

- 已完成内容；
- 已验证内容；
- 尚未完成内容；
- 当前风险和下一步。

无法清楚描述当前状态时停止继续修改。

## 11. 遵守工程约定

- 代码规范以`docs/development/02-engineering-standards.md`为准；
- 测试门禁以`docs/development/03-test-strategy-and-quality-gates.md`为准；
- 业务规则以已确认PRD/TRD为准；
- 认为约定有问题时先提出，不静默创建另一套模式。

## 12. 失败必须可见

- 审计、权限、迁移或测试失败不能伪装成功；
- 不吞异常；
- 不静默跳过测试；
- 不用通知成功代替业务成功；
- 不宣称未运行的测试通过。

## 13. 项目不可变约束

- 主干固定为提案→立案→立项→开发→上市→运营/退市；
- 产品机会资产、项目实例和产品资产保持独立；
- 产品—版本—SKU—渠道统一主模型不得分叉；
- 重大阶段门固定为4个；
- 项目当前组长是唯一A，原子任务只有一个R；
- 正式状态只能通过后端应用服务迁移；
- 权限采用RBAC + ABAC + 审计，默认拒绝；
- 平台管理权与业务数据访问权分离；
- 文件、产品版本、模板、决策和快照不可覆盖；
- Redis和Celery不保存唯一业务事实；
- 旧原型不得成为正式依赖。

## 14. 文档阅读顺序

1. `README.md`；
2. `docs/development/00-development-readiness-baseline.md`；
3. 对应PRD；
4. `docs/trd/00-system-master-trd.md`；
5. 对应领域TRD；
6. `docs/development/02-engineering-standards.md`；
7. 当前阶段任务级实施计划。

## 15. 完成前验证

至少执行与变更相关的：

```text
格式检查
静态检查
类型检查
MySQL迁移检查
相关单元和集成测试
前端类型和单元测试
OpenAPI契约检查
必要的E2E或恢复检查
```

最终报告必须区分：

- 当前实际运行并通过的验证；
- 文档记录但本轮未运行的验证；
- 被环境阻塞的验证。
